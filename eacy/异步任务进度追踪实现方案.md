# 异步任务进度追踪实现方案

> [!summary]
> 当前项目已经有 Celery worker、`extraction_jobs`、`extraction_runs`、`documents.ocr_status`、`documents.meta_status`，但进度仍主要停留在 `0/100` 和单一状态字段。要让电子病历夹抽取、靶向抽取、科研项目 CRF 抽取、OCR、metadata 等长耗时任务在界面上可追踪，需要新增统一的任务进度事件表和聚合查询接口，前端再用轮询或 SSE 渲染任务队列、阶段进度、失败原因和关联对象。

## 当前代码结论

### 已有能力

- `backend/app/workers/celery_app.py` 已定义 `ocr`、`metadata`、`extraction` 三类队列。
- `backend/app/workers/extraction_tasks.py` 已能通过 Celery 执行 `process_extraction_job(job_id)`。
- `backend/app/workers/ocr_tasks.py` 和 `metadata_tasks.py` 已分别包装文档 OCR 与 metadata 任务。
- `backend/app/models/extraction_job.py` 已有 `status`、`progress`、`error_message`、`started_at`、`finished_at`。
- `backend/app/models/extraction_run.py` 已有 run 级状态、输入快照、原始输出、解析输出、校验状态和错误信息。
- `POST /api/v1/patients/{patient_id}/ehr/update-folder` 会批量创建患者 EHR/靶向抽取任务。
- `POST /api/v1/projects/{project_id}/patients/{project_patient_id}/crf/update-folder` 会批量创建科研 CRF 抽取任务。
- 前端已有 `frontend_new/src/utils/taskStore.js` 本地任务缓存和 `getCrfExtractionProgress()` 查询单个 extraction job 的雏形。

### 主要缺口

- `ExtractionService._process_job()` 只在开始设置 `progress = 0`，完成设置 `progress = 100`，中间没有阶段进度。
- OCR 和 metadata 任务只写 `documents.ocr_status/meta_status`，不进入 `extraction_jobs`，无法统一展示在同一个任务中心。
- 批量任务接口只返回 `job_ids` 和数量，没有批次 ID，前端无法稳定展示“本次更新电子病历夹整体进度”。
- `useExtractionProgressSSE.js` 目前是空实现，没有连接后端事件流。
- `getProjectExtractionTasks()`、`getActiveExtractionTask()`、`cancelCrfExtraction()` 等前端 API 仍是空返回。
- 任务状态缺少 `stage`、`stage_label`、`current_step`、`total_steps`、`message`、`heartbeat_at` 等界面友好字段。
- 无法区分“排队中”“worker 已接收”“外部 OCR/LLM 调用中”“校验修复中”“落库中”“等待文档 OCR 完成”等不同运行阶段。

## 推荐目标形态

### 统一任务模型

新增统一任务进度表，不替代业务表，而是作为界面追踪层：

```text
async_task_batches
  └── async_task_items
        └── async_task_events
```

`extraction_jobs`、`documents` 仍保留各自业务状态；统一任务表负责跨业务的进度显示、批量聚合、事件流和用户任务中心。

### 任务类型

建议统一枚举：

| task_type | 对应业务 | 当前落点 |
|---|---|---|
| `document_ocr` | 文档 OCR | `documents.ocr_status` |
| `document_metadata` | 文档 metadata 抽取 | `documents.meta_status` |
| `patient_ehr_folder_extract` | 电子病历夹批量更新 | 多个 `extraction_jobs` |
| `patient_ehr_targeted_extract` | 患者 EHR 靶向抽取 | 单个或多个 `extraction_jobs` |
| `project_crf_folder_extract` | 科研项目 CRF 批量更新 | 多个 `extraction_jobs` |
| `project_crf_targeted_extract` | 科研项目 CRF 靶向抽取 | 单个或多个 `extraction_jobs` |
| `project_dataset_export` | 科研项目导出 | 后续导出 job |

### 状态机

统一对前端暴露这些状态：

```text
created -> queued -> running -> succeeded
                         ├── failed
                         ├── cancelled
                         └── waiting
```

说明：

- `created`：DB 已创建，尚未入队。
- `queued`：已发送到 Celery。
- `running`：worker 开始执行。
- `waiting`：等待前置条件，例如文档 OCR 尚未完成。
- `succeeded`：业务完成。
- `failed`：任务失败，有可读错误。
- `cancelled`：用户取消或业务主动停止。

## 后端改造步骤

1. 新增 `AsyncTaskBatch`、`AsyncTaskItem`、`AsyncTaskEvent` model、repository 和 migration。
2. 新增 `TaskProgressService`，提供 `create_batch()`、`create_item()`、`mark_queued()`、`update_progress()`、`mark_succeeded()`、`mark_failed()`、`aggregate_batch()`。
3. 在 `ExtractionService.create_and_process_job()`、`_create_pending_planned_job()` 和批量 folder update 中创建 task item 或 batch。
4. `_enqueue_extraction_task()` 保存 Celery task id 到 `async_task_items.celery_task_id`，同时把 item 标为 `queued/progress=5`。
5. 在 `ExtractionService._process_job()` 的关键阶段调用 `TaskProgressService.update_progress()`，同步更新 `extraction_jobs.progress`。
6. 在 OCR 和 metadata service 中也接入同一套 progress service，避免文档任务散落在 `documents` 表状态里。
7. 新增聚合接口：
   - `GET /api/v1/tasks?scope_type=patient&patient_id=...`
   - `GET /api/v1/tasks/{task_id}`
   - `GET /api/v1/task-batches/{batch_id}`
   - `GET /api/v1/task-batches/{batch_id}/events?after_id=...`
   - `GET /api/v1/tasks/stream?batch_id=...` 或 `GET /api/v1/tasks/stream?scope_type=patient&patient_id=...`
8. 修改 `GET /api/v1/extraction-jobs/{job_id}`，保留现有响应，同时可附加 `task_item` 或让前端通过 `extraction_job_id` 查询统一任务。
9. 给长时间 `running` 但 `heartbeat_at` 超过阈值的任务标记为 `stalled` 或在界面提示“可能已中断”，不要自动改业务状态。

## 前端改造步骤

1. 实现 `frontend_new/src/hooks/useExtractionProgressSSE.js`，连接后端 SSE；第一阶段也可用 2-3 秒轮询。
2. 扩展 `frontend_new/src/utils/taskStore.js`：保存 `batch_id`、`task_id`、`task_type`、`scope_type`、`related_ids`、`last_status`，页面刷新后能恢复追踪。
3. 实现 `getProjectExtractionTasks()`、`getActiveExtractionTask()`、`cancelCrfExtraction()` 等真实 API。
4. 在患者详情页、科研项目患者 CRF 页、文档列表页右上角显示当前 scope 的任务入口。
5. 批量更新返回 `batch_id` 后，界面展示批次级进度：
   - 总任务数、成功数、失败数、运行数。
   - 当前正在处理的文档/表单。
   - 可展开查看每个子任务。
6. 单个靶向抽取展示 item 级进度：
   - 当前阶段。
   - 文档名、目标表单。
   - 失败原因和重试按钮。
7. 全局 `NotificationBell` 接入任务摘要，只提示失败、完成和需要用户处理的任务，避免刷屏。

## 界面展示建议

### 患者电子病历夹

- 顶部显示“病历夹更新中 3/12”，用批次进度条表达整体进度。
- 左侧分类树或字段组旁显示抽取状态小标记：`等待中`、`抽取中`、`已更新`、`失败`。
- 抽取完成后刷新 EHR 数据；失败项保留“重试失败任务”入口。

### 科研项目 CRF

- 项目患者详情页显示当前 CRF 抽取批次。
- 每个 `target_form_key` 对应一个子任务，前端可把“正在抽取哪个访视/表单”显示出来。
- 数据集总览页只显示聚合结果，不展示所有事件。

### 文档列表

- 文档卡片把 OCR、metadata、抽取状态合并展示：
  - OCR 中。
  - 元数据抽取中。
  - 已归档，等待 EHR 抽取。
  - EHR/CRF 抽取失败。

## 第一阶段最小可落地方案

如果要快速改善界面，先做这几个点：

1. 只新增 `async_task_items` 和 `async_task_events`，`batch` 可先用 `batch_id` 字段轻量实现。
2. `extraction_jobs` 每个关键阶段更新 `progress/stage/message`，同时写 event。
3. 批量接口返回 `batch_id + job_ids`。
4. 前端用轮询 `GET /api/v1/task-batches/{batch_id}`，不先做 SSE。
5. OCR 和 metadata 暂时通过文档状态聚合进同一个接口，第二阶段再完全统一入 task item。

这样能最快解决“界面无法追踪长耗时抽取”的核心痛点，同时不破坏现有抽取落库链路。

## 最小验收单元

> [!important]
> 最小验收单元只验收“一个批量抽取操作可被创建、追踪、刷新恢复、完成/失败展示”。不要求第一阶段完成 SSE、OCR/metadata 完全统一、全局通知中心、Celery revoke 或复杂任务中心。

### 验收目标

用户在患者 EHR 或科研 CRF 页面点击一次“更新病历夹/更新 CRF”后，界面必须能看到：

- 已创建多少个后台抽取任务。
- 当前整体进度百分比。
- 当前正在处理的文档或目标表单。
- 每个子任务的状态、阶段、进度。
- 失败任务的错误原因。
- 页面刷新后仍能恢复该批次进度。
- 全部完成后 EHR/CRF 数据自动刷新或出现明确的“刷新数据”动作。

### 后端最小验收

1. 数据库已新增最小任务进度存储。
   - 可接受方案：`async_task_items` + `async_task_events`，并在 `async_task_items` 上保留 `batch_id`。
   - `async_task_items` 至少包含：`id`、`batch_id`、`task_type`、`status`、`progress`、`stage`、`stage_label`、`message`、`extraction_job_id`、`patient_id`、`project_id`、`project_patient_id`、`document_id`、`target_form_key`、`error_message`、`created_at`、`updated_at`。
   - `async_task_events` 至少包含：`id`、`batch_id`、`item_id`、`event_type`、`status`、`progress`、`stage`、`message`、`created_at`。

2. 批量抽取接口返回批次信息。
   - `POST /api/v1/patients/{patient_id}/ehr/update-folder` 返回 `batch_id`、`job_ids`、`created_jobs`、`submitted_jobs`。
   - `POST /api/v1/projects/{project_id}/patients/{project_patient_id}/crf/update-folder` 返回 `batch_id`、`job_ids`、`created_jobs`、`submitted_jobs`。
   - 当没有可创建任务时，也要返回稳定结构：`batch_id` 可为空或为本次空批次，`created_jobs = 0`，并给出可读 `message`。

3. 每个抽取 job 都能关联到 task item。
   - `async_task_items.extraction_job_id = extraction_jobs.id`。
   - `extraction_jobs.progress` 与 `async_task_items.progress` 在关键阶段保持一致。
   - 已入队任务状态为 `queued`，worker 开始后为 `running`，完成后为 `succeeded`，失败后为 `failed`。

4. 至少实现这些阶段更新。

| progress | stage | 验收点 |
|---:|---|---|
| 5 | `queued` | API 返回后数据库可查到队列状态 |
| 10 | `worker_started` | Worker 消费任务后状态变为运行中 |
| 30 | `load_document` | 已读取文档或准备抽取输入 |
| 45 | `call_extractor` | 进入 AI/规则抽取阶段 |
| 90 | `persist_values` | 开始写入候选值 |
| 100 | `completed` | job 和 item 都完成 |

5. 聚合查询接口可用。
   - `GET /api/v1/task-batches/{batch_id}` 返回批次聚合状态。
   - 响应至少包含：`batch_id`、`status`、`progress`、`total_items`、`running_items`、`succeeded_items`、`failed_items`、`items[]`。
   - `items[]` 至少包含：`task_id`、`status`、`progress`、`stage_label`、`message`、`document_id`、`target_form_key`、`extraction_job_id`、`error_message`。

6. 失败可追踪。
   - 构造一个抽取失败场景后，`extraction_jobs.status = failed`。
   - 对应 `async_task_items.status = failed`。
   - `error_message` 有内容，前端可直接展示。
   - `async_task_events` 至少有一条 `event_type = error` 的记录。

### 前端最小验收

1. 患者 EHR 页面能展示批量更新进度。
   - 点击“更新电子病历夹”后，页面出现批次进度条。
   - 进度条展示 `已完成/总数` 和百分比。
   - 子任务列表能看到文档或表单、阶段、状态。
   - 全部完成后页面刷新 EHR 数据或提供明确刷新入口。

2. 科研项目 CRF 页面能展示批量更新进度。
   - 点击“更新 CRF”后，展示同样的批次进度。
   - 子任务能展示 `target_form_key`，让用户知道正在抽取哪个表单。
   - 失败子任务能显示错误原因和对应表单/文档。

3. 页面刷新后能恢复追踪。
   - 前端把 `batch_id` 写入 `taskStore` 或页面状态存储。
   - 刷新页面后继续轮询 `GET /api/v1/task-batches/{batch_id}`。
   - 若批次已完成，展示最终状态，而不是丢失任务。

4. 轮询行为可控。
   - 第一阶段使用 2-3 秒轮询即可。
   - 批次进入 `succeeded`、`failed`、`cancelled` 后停止轮询。
   - 页面卸载时停止轮询，避免重复请求。

### 最小验收测试用例

| 编号 | 场景 | 操作 | 通过标准 |
|---|---|---|---|
| A1 | 患者 EHR 批量抽取 | 在有已归档文档的患者页点击更新病历夹 | 返回 `batch_id`，界面展示批次进度和子任务 |
| A2 | 科研 CRF 批量抽取 | 在项目患者 CRF 页点击更新 CRF | 返回 `batch_id`，子任务展示目标表单 |
| A3 | 单个任务进度推进 | 观察一个 job 从入队到完成 | 后端接口能看到 `5 -> 10 -> 30 -> 45 -> 90 -> 100` 中至少 4 个阶段 |
| A4 | 失败展示 | 构造 schema/context/document mismatch 或 mock extractor 抛错 | 批次显示失败数，子任务显示错误原因 |
| A5 | 刷新恢复 | 批次运行中刷新浏览器 | 页面能恢复同一个 `batch_id` 的进度 |
| A6 | 完成后数据可见 | 等待批次完成 | EHR/CRF 页面能看到新写入候选值或 current 值 |

### 不纳入最小验收

- SSE 实时推送。
- 全局任务中心完整页面。
- OCR 和 metadata 完全迁移到统一任务表。
- Celery 正在执行任务的强制中断。
- 多用户协同任务广播。
- 历史任务长期归档和清理策略。

## 风险与注意点

- 不要只依赖 Celery result backend。Redis result 不是业务事实来源，任务进度必须落 DB。
- 不要让前端自己推断批次进度。批量任务的成功/失败/取消需要后端聚合。
- 不要把所有中间日志都塞进 `extraction_jobs.input_json` 或 `extraction_runs.parsed_output_json`，否则查询和清理都会困难。
- `progress` 必须单调递增；失败时保持最后进度，状态改为 `failed`。
- Worker 写进度要短事务，避免外部 LLM 调用期间占用 DB 事务。
- 取消任务第一阶段只保证“不再继续落库”；对已经发出的外部 OCR/LLM 请求可以后续再做 revoke/abort。
