# 安全验收

> 本文档列出 EACY 上线前必须可手动复现的安全相关检查项。重在**验证机制是否生效**，不重复编写业务用例（业务用例见 [[用户系统与权限/验收要点]] 与各业务域 `验收要点.md`）。

## 一、鉴权（JWT 生命周期与未授权访问）

> 参考 [[用户系统与权限/验收要点]] 第 2 节。

- [ ] **未授权访问**：`ENABLE_AUTH=true` 时，不带 `Authorization` 头访问任意业务接口（如 `GET /api/v1/patients`、`GET /api/v1/documents`）→ 401 `Missing authorization token`。
- [ ] **伪造 token**：携带签名错误的 JWT 访问业务接口 → 401 `Invalid authorization token`。
- [ ] **过期 token**：access_token 过期后调业务接口 → 业务接口首次返回 401 → 前端 `request.js` 自动调 `/auth/refresh` → 重放成功（用户无感）。
- [ ] **refresh 也失效**：refresh_token 也失效后 → 自动跳 `/login`，`localStorage` 的 access/refresh/user_info 三件套被清空。
- [ ] **并发刷新单例**：N 个业务请求同时遇到 401 → `/auth/refresh` 只被调用 1 次（`refreshPromise` 单例）。
- [ ] **Admin 守门**：普通 `role=user` 调 `GET /api/v1/admin/extraction-tasks` → 403 `Admin permission required`；`role=admin` → 200。

> [!warning] 已知缺口（仅作声明）
> `/auth/logout` 当前为 no-op，无服务端会话失效；如客户端需"立即下线"，必须额外引入 refresh token 黑名单。

## 二、用户作用域隔离（A 用户看不到 B 的数据）

> 参考 [[用户系统与权限/验收要点]] 第 5 节、[[病例管理/验收要点]] 用例 2、[[文档与OCR/验收要点]] 用例 7。

- [ ] **病例隔离**：用户 A 创建病例 P；用户 B `GET /api/v1/patients` 不含 P；`GET /api/v1/patients/{P.id}` 返回 404（不是 403、不是 200）。
- [ ] **文档隔离**：用户 A 上传文档 D；用户 B `GET /api/v1/documents/{D.id}` 返回 404；`GET /api/v1/documents/v2/tree` 看不到 D 的计数。
- [ ] **项目隔离**：用户 A 创建项目 R；用户 B `GET /api/v1/research/projects` 不含 R。
- [ ] **缓存隔离**：账号 A 刷新了归档树缓存，账号 B 的缓存不应被串改（见 [[病例管理/验收要点]] 用例 8）。
- [ ] **仓储层防穿透**：尝试用 A 的 token 携带 B 的资源 ID 调任何 PATCH / DELETE → 一律 404（不是先读取再校验）。

> 当 `ENABLE_AUTH=false`（开发模式）时，`dev_admin` 会绕过 owner 过滤看到全部数据，这是预期行为。上线环境必须 `ENABLE_AUTH=true`。

## 三、敏感配置（API key 不出现在日志、不出现在前端 bundle）

- [ ] **后端日志不含 secret**：检查 API / Worker 日志，不应出现：
  - `OPENAI_API_KEY` / `LLM_API_KEY` 的实际值
  - `TEXTIN_API_KEY` / `TEXTIN_SECRET` 的实际值
  - OSS / S3 的 `AccessKeySecret`
  - JWT secret key
  - DB 密码
- [ ] **错误响应不泄密**：故意触发 LLM / OCR 错误，响应体不应回显 API key 或完整下游 URL with key。
- [ ] **前端 bundle 不含 secret**：构建前端后，对 `dist/` 下产物 grep 常见敏感关键字（`sk-`、`accessKey`、`OPENAI_API_KEY`），应无命中。
- [ ] **`.env` 不入库**：检查 `.gitignore` 包含 `.env*`；`git log -- .env` 无历史泄漏。
- [ ] **运维移交清单中明确**：API key 等不写在交付文档里，参考 [[06_集成与外部依赖/README]] 的"注意"小节。

## 四、文件上传校验（大小、类型）

> 参考 [[业务流程-文档上传与存储]] 与 [[文档与OCR/验收要点]]。

- [ ] **类型限制**：上传非 PDF / 非图片类型（如 `.exe` / `.html` / `.zip`）应被拒绝（4xx），不应静默接受。
- [ ] **大小限制**：上传超过单文件大小上限的文件应被拒绝（4xx），且不会把半截内容写入 OSS。
- [ ] **空文件**：上传 0 字节文件应被拒绝或正确进入 `skipped` 终态，不会卡住 OCR worker。
- [ ] **同名文件**：上传同名文件按业务规则处理（重命名 / 拒绝 / 覆盖，详见业务概述）；不会写入 OSS 同 key 导致互相覆盖。
- [ ] **路径注入**：上传时携带异常文件名（如 `../../etc/passwd`、含 null 字节、超长 unicode）应被规范化或拒绝，不应影响 OSS 存储路径。
- [ ] **OCR 容错**：上传图片型 PDF 触发 OCR 失败 → `ocr_status=failed`，错误信息可见，可重抽（[[文档与OCR/验收要点]] 用例 3）。

## 五、注入 / XSS

- [ ] **后端 SQL 注入防护**：所有数据库访问通过 SQLAlchemy ORM 参数绑定，**不使用 f-string 拼 SQL**。可在代码中 grep 是否存在 `f"SELECT ... {var}"` 类模式。
- [ ] **后端命令注入**：不使用 `os.system` / `subprocess.shell=True` 拼接用户输入。
- [ ] **前端 XSS 默认防护**：React + antd 默认对 `{}` 插值做 HTML 转义；不使用 `dangerouslySetInnerHTML` 渲染用户输入；如个别地方使用，必须经过 sanitize（如 `DOMPurify`）。
- [ ] **OCR / metadata 文本回显**：抽取出来的字段值（含从 LLM 返回的文本）在前端展示时，确认不会作为 HTML 渲染。
- [ ] **路径参数**：前端拼 URL 时对 `patient_id` / `document_id` 等 UUID 走 `encodeURIComponent`，避免歧义。
- [ ] **OpenAPI 文档接口**：生产部署应**关闭** `/docs` 与 `/redoc`，或加 admin 鉴权（参考 `backend/app/server.py` 配置）。

## 六、依赖安全

- [ ] **后端依赖审计**：在 CI 或定期任务中跑 `pip audit` / `safety check`，无 high/critical 未处理告警。
- [ ] **前端依赖审计**：在 CI 或定期任务中跑 `npm audit` / `pnpm audit`，无 high/critical 未处理告警。
- [ ] **依赖锁定**：`requirements.txt` / `package-lock.json`（或 `pnpm-lock.yaml`）入库，版本明确。
- [ ] **基础镜像（如使用 Docker）**：定期 rebuild，跟随上游安全补丁。
- [ ] **执行节奏建议**：随版本发布跑一次；高危 CVE 出现时按需补跑。

## 七、OSS 签名 URL 时效性

> 参考 [[文档与OCR/验收要点]] 用例 8。

- [ ] **过期生效**：`GET /documents/{id}/preview-url?expires_in=60` 拿到 URL，60s 内可访问；超过 60s 后访问返回 403 / SignatureDoesNotMatch（OSS 侧）。
- [ ] **上限强制**：`expires_in` 超过 86400 应被后端拒绝或裁剪到上限。
- [ ] **跨用户串号**：用户 A 拿到的预览 URL 不应能被推出去访问用户 B 的文档（URL 中的 path 是 A 的 OSS key，不会落到 B 的 key）。
- [ ] **owner 校验**：触发签名 URL 接口本身已带 owner 隔离（A 不能为 B 的文档拿 URL）。
- [ ] **已知限制**：当前签名 URL 不绑定 IP / UA，拿到 URL 的人在过期前都可访问；TBD 是否需加强（参考 [[文档与OCR/验收要点]] 用例 8 备注）。

## 八、其它必查项

- [ ] **CORS 配置**：生产环境 `CORS_ORIGINS` 收敛到具体前端域名，不用 `*`。
- [ ] **HTTPS**：生产部署必须经反向代理强制 HTTPS；JWT 与 OSS 签名 URL 仅经 HTTPS 流转。
- [ ] **Cookie / Storage 标记**：若使用 cookie 存 token，需 `HttpOnly + Secure + SameSite`；当前实现把 token 存 `localStorage`，需在 [[关键设计-鉴权与作用域]] 中明确风险。
- [ ] **日志保留**：审计日志（`field_value_event` 等）不会因业务删除被级联清空，软删保留审计链（见 [[病例管理/验收要点]] 用例 5）。
- [ ] **错误信息粒度**：登录失败统一 401，不区分"用户不存在 / 密码错 / 账号禁用"，避免账号枚举（见 [[用户系统与权限/验收要点]] 第 1 节）。

## 九、不通过的处理

- 鉴权 / 隔离失效：复核 `backend/app/core/auth.py`、各 Repository 的 `owner_id` 过滤是否齐全。
- secret 泄漏：立即吊销并轮换；检查 logger / exception 中是否打印了请求体或 env。
- 文件上传问题：复核 `DocumentService.upload` 与前端 `documents` API 的 MIME / size 校验。
- 依赖告警：参考各依赖的 CVE 修复版本；无补丁时评估是否替换或加 WAF 兜底。
