# 验收范围

> 本文档定义本次 EACY 交付的功能边界、运维移交内容、不在范围内事项与验收前置条件。具体可执行的验收用例分散在各业务域的 `验收要点.md` 中，由 [[功能验收清单]] 聚合。

## 一、包含的业务能力

本次交付覆盖以下七个业务域，每个域的端到端可用性由各自的验收要点定义：

| 业务域 | 验收要点入口 | 核心交付能力 |
|---|---|---|
| 病例管理 | [[病例管理/验收要点]] | 病例 CRUD、owner 隔离、列表筛选、`extra_json` 合并、软删除级联、活跃抽取状态批量查询 |
| 文档与 OCR | [[文档与OCR/验收要点]] | 上传至 OSS、自动 OCR 与 Metadata、未归档分组与一键归档、失败重抽、签名预览 URL |
| AI 抽取 | [[AI抽取/验收要点]] | 单文档抽取、证据归因、字段值历史、批量抽取进度、失败重试、Schema 字段规划、完成态保护 |
| Schema 模板与 CRF | [[Schema模板与CRF/验收要点]] | 模板设计与发布、版本不变性、病例自动绑定最新 ehr published、项目绑定指定版本、归档与软删 |
| 科研项目与数据集 | [[科研项目与数据集/验收要点]] | 项目创建与模板绑定、病例纳入与撤回幂等、project_crf 上下文与 EHR 上下文隔离、版本切换不破老数据、xlsx 导出 |
| 用户系统与权限 | [[用户系统与权限/验收要点]] | 登录注册、JWT 刷新链路、`ENABLE_AUTH` 开关、admin 守门、数据作用域隔离 |
| 管理后台 | [[管理后台/验收要点]] | 异步任务列表筛选、详情下钻、停滞探测、与业务页同步、SSE 实时推送 |

跨域横向验收见：
- [[性能验收]] —— 关键场景性能指标与验证方法
- [[安全验收]] —— 鉴权、隔离、敏感配置、上传校验、注入防护

## 三、不在范围内

参考 [[系统定位与价值]] 的"不在范围内的事项"部分，**本次交付明确不包含**：

### 业务能力维度

- **病种知识图谱推理 / CDSS 决策支持** —— 本系统只做"文本 → 字段"抽取，不做临床判断。
- **HIS / EMR 实时双向集成** —— 当前仅支持离线文档上传。
- **患者隐私自动脱敏** —— 默认数据已合规授权用于科研，系统不做去标识化。
- **多中心数据同步 / 跨机构联邦** —— 当前为单实例部署。
- **正式的细粒度权限点位** —— `require_permissions(...)` 工厂未被业务路由消费，当前只有 `role=admin` 守门与 owner 作用域隔离。详见 [[用户系统与权限/验收要点]]"已知缺口"。
- **"立即下线"会话失效** —— `/auth/logout` 当前为 no-op，无 refresh token 黑名单。
- **邮箱验证码 / 密码重置 / 微信扫码登录** —— 前端 UI 存在但后端无对应实现。
- **嵌套字段自动建 RecordInstance** —— 当前依赖 `field_path` 末端下标区分多实例，见 [[关键设计-嵌套字段与RecordInstance]]。
- **批次级取消、抽取心跳卡死告警** —— 见 [[关键设计-异步任务进度追踪]] TBD 项。

### 工程交付维度

- **单元测试用例本身** —— 仅作为辅助证据存在于 `backend/tests/`，不作为验收交付物。
- **自动化测试报告 / CI 流水线产物** —— 由 CI 工具维护，不在文档库内。
- **内部代码审查记录、迭代日志、会议纪要** —— 那是内部 `eacy/` vault 的职责。
- **性能压测的具体数值** —— 本交付提供验证方法与目标量级范围，不承诺具体压测数据，见 [[性能验收]]。

## 四、验收前置条件

### 4.1 环境准备

- 一套完整运行的 EACY 后端（FastAPI + Celery + Redis + 数据库）+ 前端（已 build 或 dev server）。
- 已配置好以下外部依赖凭据（环境变量），且能联通：
  - 对象存储（OSS / S3 / 本地磁盘）
  - TextIn OCR API key 与 endpoint
  - LLM Provider API key 与 base_url
- 数据库已执行最新 Alembic migration（与 `last_verified_commit: 132a529` 对齐）。
- `ENABLE_AUTH=true` 用于鉴权与隔离验收；`ENABLE_AUTH=false` 用于联调验收（参考 [[用户系统与权限/验收要点]] 第 3 节）。

### 4.2 账号准备

- 至少两个普通用户账号 A、B（用于验证 owner 隔离）。
- 至少一个 `role=admin` 账号（用于管理后台验收）。
- 账号能正常拿到 access_token + refresh_token。

### 4.3 数据准备

- 至少一个已发布（`status=published`）的 `ehr` 域 SchemaTemplateVersion。
- 至少一个已发布的 `crf` 域 SchemaTemplateVersion（用于科研项目绑定）。
- 测试 PDF 若干：
  - PDF A：包含可识别患者姓名 + 身份证号的扫描件
  - PDF B：内容空白或全图无文字
  - PDF C：包含出院带药清单（多条记录，用于嵌套字段抽取）
- 各业务域 `验收要点.md` 中的"前置数据"小节列出更细粒度的要求，逐条对照执行。

### 4.4 工具准备

- API 调试工具（curl / Postman / Insomnia / HTTPie 任一），用于直接打接口。
- 数据库客户端（连到目标 DB），用于执行验收用例中的 `SELECT` 验证语句。
- 浏览器（用于前端 UI 验收）。
- 一个可控的"断网 / 错凭据"方式（用于失败重试验收，见 [[AI抽取/验收要点]] 用例 5）。

## 五、验收流程建议

1. 先跑 [[安全验收]]：鉴权与隔离失败会让其它用例的数据归属失去意义。
2. 跑各业务域的 `验收要点.md`，可通过 [[功能验收清单]] Bases 视图按域筛选。
3. 跑 [[性能验收]]：在功能可用基础上验证关键场景的耗时量级。
4. 不通过项参考各域 `验收要点.md` 末尾的"不通过的处理"小节，回到对应代码路径与日志定位。
