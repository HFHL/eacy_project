# 性能验收

> 本文档列出 EACY 关键场景的**目标量级**、**验证方法**与**通过标准**。本交付**不承诺具体压测数据**，所有数字以"量级范围"形式给出，由验收方在目标环境复跑后填入实际值。

> [!warning] 影响性能的外部变量
> OCR / LLM 耗时高度依赖 TextIn / LLM Provider 的当时网络与配额。本表中的"目标量级"是常规情况下的合理预期，**任何具体数字都必须在目标环境实测后才能写入合同/承诺**。

## 一、单文档 OCR 耗时

- **场景**：用户上传一份 5~20 页的 PDF，自动入队 OCR。
- **目标量级**：单页几百毫秒到数秒级；整份 PDF 通常 10s~60s 量级（强依赖 TextIn 当时延迟）。
- **验证方法**：
  1. 选用一份典型测试 PDF（如 [[验收范围]] 中的 PDF A，10 页左右）。
  2. `POST /api/v1/documents` 上传不带 `patient_id`，记录入队时间。
  3. 轮询 `GET /documents/{id}`，记录 `ocr_status` 从 `queued → running → completed` 的时间戳。
  4. 重复 5~10 次，记录均值与 P95。
- **通过标准**：
  - [ ] 同样的 PDF 在网络稳定时，端到端 OCR 完成时间稳定在一个**可解释范围**内，无失败、无超时。
  - [ ] OCR 完成后 `parsed_data.pages.length` 与 PDF 实际页数一致；`ocr_text` 非空且包含已知关键字。
  - [ ] 失败率（OCR 步骤）在本批次内 ≤ 5%（高于此值须诊断 TextIn 配额或网络）。

## 二、单次抽取（一个 form_key）耗时

- **场景**：对一份已 OCR 的文档，针对某个 form（如"出院记录"主表）触发一次 ExtractionJob。
- **目标量级**：数秒到数十秒量级，主要由 LLM 单次调用耗时决定。
- **验证方法**：
  1. 选 1 份已 OCR 的目标文档；前端 `PatientDetail/DocumentsTab` 触发"抽取"。
  2. 记录 ExtractionJob 入队到 `status=completed` 的耗时。
  3. 重复 5 次，记录均值与 P95。
- **通过标准**：
  - [ ] 5 次抽取中 ≥ 4 次在合理量级内完成（无 retry 介入）。
  - [ ] 完成后 `extraction_runs` 写入一行 `run_no=1, status=completed`；`field_value_evidence` 与 `field_current_value` 行数 > 0。
  - [ ] 失败时 Celery 自动重试 ≤ 3 次（见 [[AI抽取/验收要点]] 用例 5）。

## 三、病例 EHR 批量更新（典型 10 文档）耗时

- **场景**：病例下有 10 份已 OCR 文档，点击"更新电子病历夹"，触发批量抽取。
- **目标量级**：数十秒到数分钟量级；Worker 并发度与 LLM 速率限制决定上限。
- **验证方法**：
  1. 准备一个病例，挂 10 份已 OCR 文档。
  2. 调 `POST /patients/{id}/ehr/update-folder`。
  3. 记录 `async_task_batch` 从 `pending → succeeded/completed_with_errors` 的耗时。
  4. 观察 `progress / succeeded_items / failed_items` 的单调演化。
- **通过标准**：
  - [ ] 批次最终落到 `succeeded` 或 `completed_with_errors`，无卡 `running` 超过 15 分钟（参见 [[管理后台/验收要点]] 停滞探测 A4）。
  - [ ] 前端 `globalBackgroundTaskPoller` 在 2~3s 内显示批次条目，progress 单调递增。
  - [ ] 失败子任务 ≤ 总数 20%（高于此值须诊断 LLM 错误模式或 Schema 配置）。

## 四、数据集导出（100 病例 × 50 字段）耗时

- **场景**：科研项目下有约 100 个 `enrolled` 病例，Schema 约 50 个叶子字段，触发 `POST .../export?scope=all`。
- **目标量级**：百级病例的导出通常在数秒到数十秒量级；瓶颈在数据库聚合与 xlsx 序列化，不在 LLM。
- **验证方法**：
  1. 在测试项目下构造 100 个 enrolled 病例，每个病例 EHR + project_crf 均有若干字段值（覆盖 ≥ 30% 字段）。
  2. 调 `POST /research/projects/{id}/export {"scope":"all", "expand_repeatable_rows": true}`。
  3. 记录响应耗时与 xlsx 字节数。
  4. 校验"患者数据"Sheet 行数 = enrolled 数；"分析宽表"展开可重复 form 后行数符合预期。
- **通过标准**：
  - [ ] 100 病例导出耗时在**单次请求超时（默认 60s）以内**，否则须改异步导出（当前未实装，列入 TBD）。
  - [ ] xlsx 文件可被 Excel / WPS 正常打开，无错列、无乱码。
  - [ ] 重复跑 3 次结果一致（同样的数据快照下）。

## 五、API 响应时间（P50 / P95）

- **场景**：日常浏览类接口的延迟分布。
- **目标量级**：
  - 读类接口（病例列表、文档树、详情）：P50 ≤ 几百毫秒，P95 ≤ 1~2 秒（取决于数据量与 ILIKE 范围）。
  - 写类接口（创建病例、PATCH 字段值）：P50 ≤ 几百毫秒。
  - 异步触发类接口（触发抽取 / 触发批量更新）：仅入队，应在数百毫秒内返回 202/200。
- **验证方法**：
  1. 用任意压测/采样工具（如 `wrk`、`ab`、`hey`、或一段 Python 脚本）对以下接口分别发起 100~500 次顺序请求：
     - `GET /api/v1/patients?page=1&page_size=20`
     - `GET /api/v1/documents/v2/tree`
     - `GET /api/v1/patients/{id}`
     - `POST /api/v1/patients`（每次用不同 name）
  2. 收集响应时间分布，计算 P50 / P95。
- **通过标准**：
  - [ ] P50 / P95 落在上面给出的量级范围内。
  - [ ] 没有 5xx；没有 timeout。
  - [ ] 在并发为 1 与并发为 10 时，P95 不出现量级跳跃。

## 六、并发承载

- **场景**：多用户同时浏览、上传、触发抽取。
- **目标量级**：当前部署形态下（单 API 进程 + 单 Worker / 双 Worker），合理目标是支持 **10~30 个活跃用户、并发上传 5~10 份文档、并发触发抽取 ≤ Celery 并发度**。具体上限由 Worker 数、LLM 配额、数据库连接池决定。
- **验证方法**：
  1. 准备 10 个测试账号，登录拿到 token。
  2. 用脚本并发发起：每账号 1 次列表 + 1 次详情 + 1 次上传 + 1 次触发抽取。
  3. 观察是否出现：
     - API 5xx
     - 数据库连接池耗尽（日志含 `QueuePool limit ... overflow`）
     - Celery 队列堆积（`async_task_batches` 大量 pending 不前进）
- **通过标准**：
  - [ ] 上述并发下无 5xx，无连接池耗尽。
  - [ ] Celery 队列在压力消除后能在数分钟内排空。
  - [ ] 任意单用户的体感延迟未显著劣化（与基线 P95 相比 ≤ 2 倍）。

## 八、不通过的处理

- OCR / LLM 耗时异常：查 TextIn / LLM 控制台的配额与延迟监控；参考 [[06_集成与外部依赖/README]]。
- API P95 劣化：查数据库慢查询日志；确认是否缺索引；ILIKE 跨多列时考虑专用索引或全文检索。
- 批量任务卡 `running`：参考 [[管理后台/验收要点]] A4 与 [[关键设计-异步任务进度追踪]]。
- 并发耗尽：调大数据库连接池 / Celery worker 数 / LLM 并发上限，参考 [[07_部署与运维/README]]。
