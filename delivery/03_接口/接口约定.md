# 接口约定

> [!info] 范围
> 本文是 EACY 后端所有 HTTP 接口的**通用约定**。各业务分组的具体语义见 [[业务API说明/Auth|Auth]]、[[业务API说明/Patients|Patients]] 等；参数表请直接看 [[OpenAPI访问|OpenAPI]]。

## 一、路径前缀

所有 v1 接口挂在 `/api/v1/` 下（见 `app/api/v1/__init__.py`）。

- 业务路由：`/api/v1/<group>/...`，例如 `/api/v1/patients`、`/api/v1/documents`
- 健康检查：`GET /api/v1/health` → `{"status": "ok"}`（不鉴权，可用于探活）

未来出现破坏性变更会另起 `/api/v2/` 前缀，**当前不存在 v2**。

## 二、鉴权机制

### 2.1 JWT Bearer Token

请求头：

```
Authorization: Bearer <access_token>
```

- 算法 `HS256`，密钥与算法由后端环境变量 `JWT_SECRET_KEY` / `JWT_ALGORITHM` 控制（见 `core/config.py`）。
- access token 通过 [[业务API说明/Auth|Auth]] 的登录接口获取，并由 refresh token 续期。
- token 解码失败、缺失 `user_id` 字段等情形会触发 401，错误响应见下文"错误响应统一格式"。

### 2.2 `ENABLE_AUTH` 开关

`core/config.py` 中的 `ENABLE_AUTH`（默认 `False`，开发环境通常关闭）：

| ENABLE_AUTH | 未带 Authorization 时 |
|---|---|
| `False`（开发默认） | 注入内置 `DEV_ADMIN_USER`（id=`dev_admin`、role=`admin`、permissions=`["*"]`） |
| `True`（生产） | 401，强制要求带合法 token |

> [!warning] 生产部署必须设置 `ENABLE_AUTH=True`
> 否则任何匿名请求都会获得管理员权限。

### 2.3 当前用户与权限

`get_current_user` 返回 `CurrentUser(id, username, role, permissions)`：

- 写操作（创建病例、上传文档、抽取任务等）在 service 层用 `uuid_user_id_or_none(current_user)` 落入 `created_by` / `uploaded_by` 列；非 UUID 形式（如 `dev_admin`）落 `NULL`。
- 管理后台路由额外用 `require_admin_user` / `is_admin_user` 做角色校验：`role == "admin"` 或 `permissions` 含 `"*"`。

### 2.4 流式接口的查询参数 token

`GET /api/v1/documents/{id}/stream` 与 `/pdf-stream` 同时接受：
- 标准 `Authorization` 头；或
- 查询参数 `?access_token=<jwt>`（用于 `<iframe>` / `<embed>` 等无法带头的场景）

详见 [[业务API说明/Documents|Documents]]。

## 三、错误响应统一格式

### 3.1 `CustomException` 抛出的业务异常

`server.py` 注册了 `CustomException` 的全局 handler，响应体固定为：

```json
{
  "error_code": "<业务错误码>",
  "message": "<可展示给用户的描述>"
}
```

HTTP status code 来自 `exc.code`（默认 400）。

### 3.2 FastAPI / Pydantic 校验错误

`HTTPException` 与 Pydantic 校验失败仍走 FastAPI 默认响应格式：

```json
{ "detail": "..." }
```

> [!info] 对接建议
> 调用方应同时识别 `error_code` 与 `detail` 两种格式。`error_code` 优先用于业务分支；`detail` 通常是参数校验/资源不存在等通用情况。

### 3.3 鉴权失败

鉴权中间件失败（缺 token、解码错误、签名错误）由 `on_auth_error` 处理：

```json
{ "error_code": null, "message": "<原因>" }
```

HTTP 401。

## 四、错误码命名规范

`core/exceptions/base.py` 定义了 `CustomException` 基类，业务侧通过子类化扩展：

```python
class CustomException(Exception):
    code = 400              # HTTP status
    error_code = "BAD_GATEWAY"
    message = "BAD GATEWAY"
```

约定：

- `error_code` 用 **大写下划线**（`SCREAMING_SNAKE_CASE`），形如 `PATIENT_NOT_FOUND`、`EXTRACTION_CONFLICT`。
- 当前代码库中大量场景仍使用 `HTTPException(status, detail=...)` 直接抛 `detail`，**尚未全量迁移到 `error_code`**。新写后端 API 推荐定义子类。
- 业务侧已显式表达的语义错误（NotFound / Conflict）见各分组文档"错误码业务含义"小节。

## 五、分页规范

列表型 GET 接口统一使用：

| 参数 | 默认 | 范围 | 含义 |
|---|---|---|---|
| `page` | `1` | `>=1` | 第几页（**1 起始**） |
| `page_size` | `20` | `1-100` | 每页条数 |

响应固定形状：

```json
{
  "items": [...],
  "total": 123,
  "page": 1,
  "page_size": 20
}
```

> [!warning] 例外
> - `/admin/extraction-tasks` 使用 `limit` / `offset`（标准游标式分页），不是 `page` / `page_size`。
> - `/task-batches/{batch_id}/events` 使用 `after_id` + `limit` 增量拉取。

## 六、ID 规范

- 主键统一为 **UUID v4 字符串**，序列化时小写带连字符（如 `c1d2…`）。
- URL path 中的 `{patient_id}` / `{document_id}` 等都接受 UUID 字符串，**大小写不敏感**（后端会做 `UUID(...)` 标准化）。
- 写操作里 `created_by` / `uploaded_by` 等列也是 UUID。非 UUID 形式（如开发态的 `dev_admin`）会落 `NULL`，详见 `uuid_user_id_or_none`。
- 软删除：大多数实体有 `deleted_at`，被软删的资源 GET 默认返回 404（除非显式查归档）。

## 七、时区

- 数据库列 `created_at` / `updated_at` 等存 UTC。
- API 响应以 ISO 8601 字符串返回，**不带时区后缀视为 UTC**。
- Celery / 业务侧的"今天 / 今日"统计以 **Asia/Shanghai** 为准（与 Celery beat 时区一致），见 [[业务流程-异步任务监控]]。

## 八、CORS

`server.py` 注册的 `CORSMiddleware`：

```python
allow_origins=["*"]
allow_credentials=True
allow_methods=["*"]
allow_headers=["*"]
```

> [!warning] 生产建议收紧
> `allow_origins=["*"]` 与 `allow_credentials=True` 共存仅在开发期可接受；生产应改为明确的前端域名白名单。

## 九、常用 HTTP 状态码语义

| 状态码 | 业务含义 |
|---|---|
| 200 | 成功（同步查询/更新） |
| 201 | 资源创建成功 |
| 202 | 已接受，异步处理中（OCR / 抽取 / 元数据等触发任务的端点） |
| 204 | 删除成功（无响应体） |
| 400 | 参数校验失败 / 业务参数非法 |
| 401 | 未鉴权 |
| 403 | 已登录但无权限（典型场景：非 admin 访问 `/admin/*`） |
| 404 | 资源不存在或已软删 |
| 409 | 状态冲突（如重复抽取、已发布的版本再编辑等） |
