# 监控与告警

> 本文给出 EACY 上线后**建议监控的核心指标**与**告警阈值参考**。具体监控平台（Prometheus / Zabbix / 云厂商监控）与告警通道（钉钉 / 飞书 / 邮件 / 短信 / Webhook）由 ops 选型，本文不固化。占位标记 (TBD) 处需由 ops 在接入时确定。

## 一、监控全景

EACY 监控分四层：

| 层级 | 关注点 |
|---|---|
| 应用层 | API 错误率、延迟、Celery 任务成功率与队列堆积 |
| 依赖层 | 数据库连接数 / 慢查询、Redis 内存与连接数、外部 API（TextIn / LLM / OSS） |
| 系统层 | CPU、内存、磁盘、网络 |
| 业务层 | 抽取任务平均耗时、待归档文档数 |

## 二、应用层关键指标

### 2.1 API 错误率与延迟

| 指标 | 含义 | 建议阈值 | 告警等级 |
|---|---|---|---|
| HTTP 5xx 比例（5 分钟滑窗） | 服务异常 | > 1% 警告；> 5% 紧急 | warning / critical |
| HTTP 4xx 比例 | 客户端 / 鉴权异常 | 突增 200% 警告 | warning |
| P95 响应时长 | 接口性能 | > 1s 警告（依业务调整） | warning |
| `/api/v1/auth/login` 失败率 | 鉴权异常 | > 30% 警告（可能密钥不匹配） | warning |

数据源：Nginx access log 或 FastAPI 中间件埋点 (TBD：当前仓库无固定 metrics endpoint)。

### 2.2 Celery 队列堆积

```bash
redis-cli -n 1 llen ocr
redis-cli -n 1 llen metadata
redis-cli -n 1 llen extraction
```

| 队列 | 健康值 | 警告阈值 | 紧急阈值 |
|---|---|---|---|
| `ocr` | 0~10 | > 50 持续 5 分钟 | > 200 |
| `metadata` | 0~10 | > 50 持续 5 分钟 | > 200 |
| `extraction` | 0~5 | > 20 持续 10 分钟 | > 100 |

持续堆积通常意味着对应 worker 未启动或被外部依赖卡住，见 [[常见故障排查#OCR任务卡queued]]。

### 2.3 Celery Worker 心跳

```bash
poetry run celery -A app.workers.celery_app.celery_app inspect ping
```

任一 worker 30 秒内无响应 → critical。`SERVER_DEPLOYMENT_NOTES.md` 记录过 `missed heartbeat` 案例，已是已知症状。

### 2.4 任务执行结果

| 指标 | 含义 | 阈值 |
|---|---|---|
| 抽取任务失败率（1 小时） | `extraction_job` 终态 = failed 占比 | > 10% 警告 |
| OCR 任务失败率 | OCR 任务终态 failed 占比 | > 5% 警告 |
| 任务平均时长 | per-queue 平均 | 突增 100% 警告 |

数据源：`extraction_job` / `document` 状态表，由监控脚本定时查询 (TBD：可在 [[科研项目与数据集]] 与 [[文档与OCR]] 模块对应表上加视图)。

## 三、依赖层关键指标

### 3.1 数据库

| 指标 | 建议阈值 |
|---|---|
| 活跃连接数 / max_connections | > 80% 警告 |
| 等待锁的事务数 | > 5 警告 |
| 慢查询（> 1s） | 单条出现即记录，5 分钟 > 10 条警告 |
| 复制延迟（如有从库） | > 30s 警告 |

> [!warning] 连接耗尽是历史已知风险
> `SERVER_DEPLOYMENT_NOTES.md` 记录过 `asyncpg.exceptions.TooManyConnectionsError`，必须把连接数监控加入告警。处置见 [[常见故障排查#数据库连接耗尽]]。

### 3.2 Redis

| 指标 | 建议阈值 |
|---|---|
| 内存使用率 | > 80% 警告 |
| 连接数 | > 80% maxclients 警告 |
| OPS（突增） | 突增 5× 警告 |

### 3.3 外部 API

| 依赖 | 监控点 |
|---|---|
| TextIn OCR | OCR worker 调用失败率、5xx、超时 |
| LLM Provider | metadata / extraction worker 调用失败率、403、超时 |
| 阿里云 OSS | API 调用失败率、CORS 错误（可由前端上报） |

| 阈值建议 | 警告 | 紧急 |
|---|---|---|
| 外部调用失败率（5 分钟） | > 5% | > 30% |
| 外部调用 P95 延迟 | TextIn > 30s；LLM > 60s | 超过对应 `*_TIMEOUT_SECONDS` 一半 |

## 四、系统层指标

由系统自带 agent（node_exporter / 云监控）采集：

| 指标 | 阈值 |
|---|---|
| CPU 5min 平均 | > 80% 警告，> 95% 紧急 |
| 内存使用率 | > 85% 警告 |
| 磁盘使用率 | > 80% 警告，> 90% 紧急 |
| Inode 使用率 | > 80% 警告 |
| 网络丢包 | > 1% 警告 |

> [!info] Celery worker 日志会持续增长
> `logs/celery.log` / `journalctl` 都需要日志轮转或保留期策略，否则磁盘易满。

## 五、业务层指标（可选）

| 指标 | SQL 来源 | 建议 |
|---|---|---|
| 待归档文档数 | `document` 表按状态聚合 | 持续 > 100 警告 |
| 待抽取任务数 | `extraction_job.status = queued` | 持续 > 50 警告 |
| 上传量 | 每日新增 `document` 行数 | 异常陡降 → 上游接入异常 |

## 六、告警通道（占位）

| 通道 | 受众 | 等级 | 配置项 |
|---|---|---|---|
| 钉钉 / 飞书机器人 | 全员 | warning + | (TBD: Webhook URL) |
| 短信 | on-call | critical | (TBD: 服务商 + 模板 ID) |
| 邮件 | 全员 | 所有 | (TBD: SMTP) |
| 电话 | on-call | critical（夜间） | (TBD: 是否启用) |

> [!warning] 别一开始就 critical 全开
> 告警疲劳比"没告警"更危险。上线第一周建议只开 critical 短信 + warning 邮件，再逐步收紧。

## 七、健康检查脚本（可选）

可在 ops 平台部署一个 5 分钟巡检任务：

```bash
#!/bin/bash
set -e
# API 探活
curl -fsS -o /dev/null http://127.0.0.1:8000/docs

# Redis 队列长度
for q in ocr metadata extraction; do
  len=$(redis-cli -n 1 llen "$q")
  echo "queue=$q len=$len"
done

# Worker 心跳
cd /opt/eacy_project/backend
poetry run celery -A app.workers.celery_app.celery_app inspect ping --timeout 5
```
