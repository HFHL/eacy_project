# 首次部署手册

## 适用场景

在一台**全新服务器**上从零部署 EACY，覆盖：依赖安装 → 配置 → 数据库初始化 → 后端 → Celery → 前端 → 健康检查。日常升级见 [[升级流程]]。

## 前置条件

- [ ] 服务器系统：Linux（CentOS / Ubuntu / Debian 任一）
- [ ] Python `3.11.7`（见 `backend/pyproject.toml`）
- [ ] Node.js `18+`
- [ ] PostgreSQL `14+`（asyncpg）或 MySQL（aiomysql），已创建库与账号
- [ ] Redis `6+`，可达
- [ ] OSS / TextIn / LLM 三组密钥已就绪
- [ ] 仓库代码已 `git clone` 到 `/opt/eacy_project`（路径可调，下文沿用此值）
- [ ] Nginx 已安装（如使用 [[部署架构]] 中的反代方案）

## 执行步骤

### 1. 安装后端依赖

```bash
cd /opt/eacy_project/backend
poetry install
```

> [!info] 也可用 uv
> `uv venv --python 3.11.7 && uv pip install -e .`，后续命令把 `poetry run` 替换为 `uv run` 即可。

### 2. 配置 `.env`

在项目根 `/opt/eacy_project/.env` 创建配置文件，**所有键的含义与默认值见 [[环境变量清单]]**。最小可运行示例参考 `DEPLOYMENT_RUNBOOK.md` 第 4 节。

> [!warning] 必须替换的占位
> `JWT_SECRET_KEY` · `OSS_*` · `TEXTIN_*` · `OPENAI_API_KEY` · `DATABASE_URL`。

### 3. 初始化数据库（执行 migration）

```bash
cd /opt/eacy_project/backend
poetry run alembic upgrade head
```

迁移会创建用户表与 5 个种子账号（详见 `DEPLOYMENT_RUNBOOK.md` §6.2）。

> [!warning] migration 不会回滚业务数据
> 仅对一张新库执行 `upgrade head`。已有库的升级见 [[升级流程]]。

### 4. 启动后端 API

#### 4.1 systemd（推荐生产）

```bash
cd /opt/eacy_project
sudo bash deploy/production/install-systemd-nginx.sh
sudo systemctl enable --now eacy-backend
sudo systemctl status eacy-backend
```

实际命令形态（由 unit 调起）：

```bash
gunicorn main:app \
  -k uvicorn.workers.UvicornWorker \
  -b 127.0.0.1:8000 \
  --workers 2 \
  --timeout 180
```

#### 4.2 调试启动

```bash
cd /opt/eacy_project/backend
poetry run python main.py --env local
```

### 5. 启动 Celery Worker（按队列分别启动）

详细参数见 [[Celery任务运维]]。`deploy/production/install-systemd-nginx.sh` 会安装三个 unit：

```bash
sudo systemctl enable --now eacy-celery-ocr
sudo systemctl enable --now eacy-celery-metadata
sudo systemctl enable --now eacy-celery-extraction
```

unit 内的等价命令：

```bash
# OCR
celery -A app.workers.celery_app.celery_app worker -Q ocr        --loglevel=info --concurrency=1
# Metadata
celery -A app.workers.celery_app.celery_app worker -Q metadata   --loglevel=info --concurrency=1
# Extraction
celery -A app.workers.celery_app.celery_app worker -Q extraction --loglevel=info --concurrency=2
```

> [!warning] 不要遗漏任何一个队列
> 没启动 metadata worker 会导致 OCR 完成后归档元数据缺失；没启动 extraction worker 会导致 EHR/CRF 抽取任务一直卡 queued。见 [[常见故障排查#OCR任务卡queued]]。

### 6. 前端构建与部署

```bash
cd /opt/eacy_project
npm install
npm run build -w xidong-crf-prototype-new
```

产物在 `frontend_new/dist`，由 Nginx 托管（`deploy/production/nginx/eacy.conf`）。

```bash
sudo nginx -t && sudo systemctl reload nginx
```

### 7. 验证健康

```bash
# API 存活
curl -I http://127.0.0.1:8000/docs

# Redis 队列（应为 0）
redis-cli -n 1 llen ocr
redis-cli -n 1 llen metadata
redis-cli -n 1 llen extraction

# Celery worker 心跳
poetry run celery -A app.workers.celery_app.celery_app inspect ping
```

## 验证方式

- [ ] `systemctl status eacy-backend eacy-celery-ocr eacy-celery-metadata eacy-celery-extraction` 全部 `active (running)`
- [ ] 浏览器打开站点，使用种子账号 `user1@example.com / 123456` 能登录
- [ ] 上传一份 PDF：状态依次走过"上传 → OCR → 元数据 → 归档"
- [ ] PDF 预览正常返回 `200 application/pdf`（**不是** 302 到 OSS）
- [ ] Nginx access 日志无 5xx

## 回滚方式

首次部署回滚 = 停服并清理。

```bash
sudo systemctl stop eacy-backend eacy-celery-ocr eacy-celery-metadata eacy-celery-extraction
sudo systemctl disable eacy-backend eacy-celery-ocr eacy-celery-metadata eacy-celery-extraction
sudo rm /etc/systemd/system/eacy-*.service
sudo rm /etc/nginx/sites-enabled/eacy.conf
sudo systemctl reload nginx
```

数据库回滚：首次部署还未承载真实数据，可直接 `DROP DATABASE` 重来。已有真实数据时见 [[备份与恢复]]。

> [!warning] 上线前最后检查
> - `ENABLE_AUTH=true`
> - `DEBUG=false`（`ENV=prod` 自动满足）
> - `JWT_SECRET_KEY` 已替换为随机值
> - 种子账号 `admin@example.com` 等已修改密码或禁用
