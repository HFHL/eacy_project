# 备份与恢复

## 适用场景

为 EACY 制定数据备份策略，并在数据丢失 / 误操作 / 升级失败时按本文恢复。本文给出**决策点与建议**，不强制具体方案——具体频率与目标存储由 ops 结合公司基础设施决定。

## 备份对象

| 数据类型 | 来源 | 重要性 | 是否易再生 |
|---|---|---|---|
| 关系型数据库 | 患者、文档、抽取任务、字段值、用户等 | 关键 | 否 |
| OSS 上的原始文档 | 用户上传的 PDF / 影印件 | 关键 | 否（原件可能用户已遗失） |
| `.env` 配置 | 密钥与连接串 | 中 | 部分可再生（密钥不可） |
| OCR markdown / metadata 缓存 | 计算产物 | 低 | 可重跑 OCR / metadata 任务再生 |

> [!info] Redis 不需备份
> Redis 仅承担 Celery broker / result backend 与业务缓存，全部内容可由 DB / 业务流程重建。

## 一、数据库备份

### 1.1 备份策略决策点

| 决策点 | 选项 | 建议（仅建议） |
|---|---|---|
| 备份方式 | `pg_dump` 逻辑备份 / WAL 物理归档 / 云厂商快照 | 中小规模优先逻辑备份；TB 级用物理归档或快照 |
| 频率 | 每天 / 每小时 / 持续 | 日全量 + WAL 增量；或云厂商每日快照 |
| 保留周期 | 7 / 30 / 90 天 | 30 天滚动 + 关键发布版本永久保留 |
| 备份位置 | 同机 / 异机 / 跨可用区 / 跨地域 | 至少异机；上线后争取跨可用区 |
| 加密 | 是 / 否 | 含患者敏感字段，建议加密 |

> [!warning] 不要把备份放在被备份的同一台机
> 否则磁盘损坏 = 备份与原库同时丢失。最低要求异机或异盘。

### 1.2 逻辑备份示例（PostgreSQL）

```bash
# 备份
pg_dump \
  --host <DB_HOST> --port <DB_PORT> \
  --username <DB_USER> \
  --format=custom \
  --file=/var/backup/eacy/eacy_$(date +%Y%m%d_%H%M%S).dump \
  <DB_NAME>
```

存储路径与权限：

```bash
sudo install -d -m 700 -o eacy -g eacy /var/backup/eacy
```

可结合 `cron` 每日执行，详细 cron 表达式由 ops 决定 (TBD)。

### 1.3 备份验证

> [!warning] 没验证过的备份等于没备份
> 每月至少做一次"恢复演练"（见下文 §3）。

## 二、OSS 备份

### 2.1 决策点

| 决策点 | 建议 |
|---|---|
| 同 Region 跨 Bucket 复制 | 阿里云 OSS 的"跨区域复制"功能可用，规则可基于前缀 `documents/` |
| 版本控制 | 启用 Bucket 版本控制，对抗误删 / 覆盖 |
| 生命周期 | 不建议为节省成本设置短期过期；原件长期保留 |

### 2.2 操作建议

由阿里云控制台或 `ossutil` 完成；本文不固化命令，避免误用：

- 启用 Bucket Versioning
- 配置 Cross-Region Replication 到容灾 region
- IAM 权限收敛，备份账号仅允许 `oss:GetObject` + `oss:PutObject` 到目标 Bucket

## 三、恢复演练

### 3.1 数据库恢复演练

每月一次，目标：**确认备份文件可用且恢复时间满足业务可接受窗口**。

```bash
# 1. 在隔离机上准备空库
createdb eacy_restore_drill

# 2. 还原
pg_restore \
  --host <DRILL_HOST> --port 5432 \
  --username <DB_USER> \
  --dbname eacy_restore_drill \
  --no-owner --no-privileges \
  /var/backup/eacy/eacy_YYYYMMDD_HHMMSS.dump

# 3. 抽样校验
psql -h <DRILL_HOST> -U <DB_USER> -d eacy_restore_drill -c "SELECT count(*) FROM patient;"
psql -h <DRILL_HOST> -U <DB_USER> -d eacy_restore_drill -c "SELECT count(*) FROM document;"
psql -h <DRILL_HOST> -U <DB_USER> -d eacy_restore_drill -c "SELECT count(*) FROM extraction_job;"
```

记录单条耗时，作为 RTO 参考。

### 3.2 真实故障恢复步骤

> [!warning] 在生产做恢复前，**先停业务**
> 否则恢复中途到达的写入会丢失或破坏一致性。

```bash
# 1. 停服
sudo systemctl stop eacy-backend eacy-celery-ocr eacy-celery-metadata eacy-celery-extraction

# 2. 备份当前损坏库快照（取证 / 二次救援用）
pg_dump --format=custom --file=/tmp/eacy_broken_$(date +%s).dump <DB_NAME>

# 3. 重建目标库
dropdb <DB_NAME>
createdb <DB_NAME>

# 4. 从最近一个有效备份还原
pg_restore --dbname <DB_NAME> --no-owner --no-privileges <最新有效备份.dump>

# 5. 若启用了 WAL 归档，按 PITR 把时间点拉到故障前
#   具体命令依赖云厂商 / 自建归档方案 (TBD)

# 6. 检查 schema 是否与代码一致
cd /opt/eacy_project/backend
poetry run alembic current

# 7. 启服
sudo systemctl start eacy-backend eacy-celery-ocr eacy-celery-metadata eacy-celery-extraction
```

### 3.3 OSS 文件恢复

- **单文件误删**：从 Bucket 历史版本恢复，或从复制 Bucket `cp` 回来
- **大量损坏**：用复制 Bucket 整目录同步回主 Bucket
- **OSS 与 DB 不一致**：以 DB `document` 表为准。DB 有记录但 OSS 无对象 → 标记为缺失或手动重传

## 四、`.env` 与密钥

- `.env` 不进 git
- 由 ops 在密码管理器（如 Bitwarden / 公司 KMS）单独保管
- JWT_SECRET_KEY 一旦泄露必须轮换，并强制所有用户重新登录
