# 文档结构说明

> 解释本 Vault 的组织原则、frontmatter 约定、Bases 索引、以及"文档与代码"如何保持同步。**对接 / 维护本套文档前请先读这一篇**。

## 一、组织原则

### 1. 属性优先，目录其次
读者通过 **`09_索引/` 下的 Bases 视图**找文档（按业务域、按状态、按受众过滤），而不是靠目录树深挖。目录是物理组织，Bases 视图是逻辑组织。

### 2. 一文一职
每篇文档只承担一个职责：
- 一个**业务流程**（如"新建病例"）
- 一个**业务 API**（如 `POST /api/v1/extraction/jobs` 的业务说明）
- 一张**数据表**（如 `extraction_job`）
- 一个**部署任务**（如"首次部署"）
- 一个**故障场景**（如"Celery 任务卡住"）

**不要把多张表合并到一篇、不要把整套 API 写在一文。**单文件超过 400 行就该拆。

### 3. 单一事实源（Single Source of Truth）
| 信息类型 | 单一事实源 | 文档里写什么 |
|---|---|---|
| API 参数、类型、状态码 | FastAPI 生成的 OpenAPI (`/docs`) | 业务含义、调用注意事项、错误码业务原因 |
| 表字段、类型、约束 | SQLAlchemy model + Alembic migration | 业务含义、生命周期、典型查询 |
| 前端路由 | `frontend_new/src/main.jsx` / `App.jsx` | 页面用途、状态流转 |
| 业务规则与决策依据 | **本 Vault** | 完整描述 |

**永远不要把字段表从代码 copy 到 md**，那样维护成本翻倍且必然发散。

## 二、Frontmatter 字段约定

每篇文档头部必须有 YAML frontmatter，Bases 视图依赖它来过滤聚合。

```yaml
---
type: business-flow | api | data-model | deploy | troubleshooting | acceptance | reference | index
module: 病例管理 | 文档与OCR | AI抽取 | Schema模板与CRF | 科研项目与数据集 | 管理后台 | 用户系统与权限 | 部署运维 | 导读
status: draft | reviewed | verified | deprecated
audience: [tech-lead, integrator, ops, reviewer]
code_path:                 # ★ 关键：本文档所描述的核心代码文件
  - backend/app/services/extraction_service.py
  - backend/app/workers/extraction_tasks.py
api_endpoints:             # 仅 api / business-flow 文档需要
  - POST /api/v1/extraction/jobs
related_tables: [extraction_job, extraction_run]
last_verified_commit: 132a529   # ★ 上次人工核对时的 git short hash
last_verified_date: 2026-05-13
owner: 后端 | 前端 | 运维 | EACY 团队
---
```

### 字段含义
- **type**：决定文档归类，Bases 会按此过滤。
- **module**：业务域名，与 `02_业务域/` 子目录名一致。
- **status**：
  - `draft` — 草稿
  - `reviewed` — 已经过内部审阅
  - `verified` — 已与最新代码对齐核对
  - `deprecated` — 已废弃但暂未删除
- **audience**：标注主要受众，便于按角色筛选。
- **code_path**：本文档描述的核心代码文件，用于"文档-代码"双向锚点。
- **last_verified_commit / last_verified_date**：人工核对过文档与该 commit 的代码一致时的时间戳；超过 30 天未更新会被 `09_索引/文档健康度.base` 标红。

## 三、文档与代码的同步机制

文档烂尾的根本原因是"代码变了文档没变"。本套文档用三条机制对抗：

### 1. 反向锚点
所有文档在 frontmatter 写 `code_path`。一个简单的脚本（`scripts/check_docs.py`，本阶段尚未实装）可以：
- 扫所有 `.md` 的 `code_path` 字段，若文件不存在 → 报告失效
- 用 `git log --since="<last_verified_date>" -- <code_path>` 查看代码是否变动 → 若变动则标记文档"待 re-verify"
- 可挂入 pre-commit hook

### 2. Bases 自动索引
`09_索引/` 下的视图（API清单、数据表清单、文档健康度）**完全由 frontmatter 自动聚合**，无需手维护汇总表。新增一篇 API 文档，frontmatter 写对了就自动出现在 API 清单里。

### 3. 月度健康度 review
每月跑一次 `09_索引/文档健康度.base`，查看 `last_verified_commit` 过期的红条目，定向 re-verify。比"定期通读全套文档"现实得多。

## 四、Obsidian 特性使用约定

| 特性 | 使用场景 |
|---|---|
| **Wikilink `[[ ]]`** | 所有跨文档引用必须用 wikilink，不要用相对路径 |
| **Callout** | `> [!info]`（提示）`> [!warning]`（破坏性变更）`> [!example]`（请求/响应样例）`> [!todo]`（待办） |
| **Properties 面板** | 编辑 frontmatter 用属性面板，避免 YAML 语法错误 |
| **Mermaid** | 时序图、流程图、ER 图——文本可 diff，优于截图 |
| **Canvas (.canvas)** | 系统架构图、模块全景图、ER 全景——可拖拽，适合可视化整体 |
| **Bases (.base)** | 索引视图——只在 `09_索引/` 下使用 |

> [!warning] 不要在文档里嵌图片
> 优先用 Mermaid / Canvas。截图易过期、不可 diff、增大仓库体积。确实需要时放 `_attachments/`。

## 五、新建文档流程

1. 从 `_templates/` 复制对应类型的模板
2. 填写 frontmatter（**所有字段都填，未知用 TBD**）
3. 写正文，引用其他文档时用 wikilink
4. 写完检查：
   - [ ] frontmatter 字段齐全
   - [ ] `code_path` 指向真实存在的文件
   - [ ] 没有把 API 参数表 / 表字段表从代码 copy 到 md
   - [ ] 引用的其他文档用 wikilink
   - [ ] 单文件 < 400 行，否则拆分

## 六、本套文档不做的事

- **不替代** FastAPI 的 OpenAPI 文档
- **不替代** 源码（含注释、类型标注）
- **不替代** Alembic migration（表结构演进）
- **不记录** 已经合并的迭代日志（看 git log）
- **不记录** 内部研发流程、会议纪要、未发布的实验（那是内部 `eacy/` vault 的职责）
