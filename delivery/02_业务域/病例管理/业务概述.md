# 业务概述

## 一句话定位

**病例（Patient）** 是 EACY 的核心业务主体，承载某一个体患者的所有文档、抽取出的结构化字段值，以及对若干科研项目的纳入关系。**病例池（PatientPool）** 是用户视角下的病例列表（按 `owner_id` 隔离）。

## 核心概念

### Patient（病例）
- 由 `name` + 一组人口学/就诊摘要字段（`gender / birth_date / age / department / main_diagnosis / doctor_name`）+ `extra_json`（脱敏字段等扩展信息）构成。
- 字段定义见 [[表-patient]]，**不在此罗列**。
- 标识为 UUID 字符串；软删除通过 `deleted_at` 实现，未填写 `deleted_at` 的视为活跃。

### PatientPool（病例池）
- 不是一张表，是**用户视角下的可见病例集合**。
- 后端 `list_patients` 通过 `owner_id` 过滤：每个用户只看到自己创建的病例（见 `PatientRepository.list_active`）。
- 顶部聚合卡片由 `list_patients_with_stats` 返回的 `page_statistics` 提供：当前页总文档数、平均完整度、当日新增数。

### 患者归属用户（owner_id）
- 创建病例时 `owner_id = uuid_user_id_or_none(current_user)`。
- 列表、详情、更新、删除均带 `owner_id` 过滤；他人创建的病例对当前用户**不可见**。
- 这是当前的**单租户多用户**隔离方式；未来若引入团队/中心维度，需扩展。

> [!info] 关于 active_state / archived
> 任务上下文里提到的 "active_state" 在当前代码中**不存在**于 `Patient` 模型。本域只有 `deleted_at`（软删除）一种状态机；"归档"语义仅对文档而言——文档通过 `document.patient_id` 归档到病例，详见 [[文档与OCR/业务概述]]。如需新增"病例归档/休眠"状态，待与团队确认。

## 关键设计

### 1. 多对多到科研项目
- 关联表 `project_patient`（见 [[表-project_patient]]），承载入组编号 `enroll_no`、入组时间 `enrolled_at`、状态 `status`。
- 一个病例可以同时被纳入多个项目；一个项目可以纳入多个病例。
- `list_patient_projects` 在患者详情返回所有关联项目，用于"关联项目"展示。

### 2. 软删除 + 业务数据级联
- `delete_patient` 不直接物理删除，而是：
  1. 软删除病例本身（`deleted_at = utcnow()`）
  2. 软删除该病例的所有文档（`document_repository.soft_delete_by_patient`）
  3. 撤回所有项目入组（`project_patient_repository.withdraw_by_patient`）
- 历史的 `field_current_value / field_value_event / field_value_evidence` **不删**，保留审计链。

### 3. 创建即初始化 EHR 上下文
- `create_patient` 默认 `initialize_ehr=True`：自动拉取 `ehr` 域的最新 `published` 版本 SchemaVersion，并调用 `EhrService.get_or_create_patient_ehr_context` 建一条 `data_context`（context_type=patient_ehr）。
- 这保证病例一被创建就**有承载抽取结果的容器**，后续抽取任务无须再判空。
- 若当前没有任何已发布的 ehr Schema，跳过此步（不会失败）。

### 4. 数据完整度计算
- `compute_patient_stats` 按"最新一条 patient_ehr 上下文"计算 `data_completeness = 已填字段数 / Schema 叶子字段总数 × 100`。
- 数组下标（如 `medications.0.name`）被规范化为 `medications.name`，避免重复实例稀释完整度。
- 是否"已填"：`value_text` 非空、或 `value_number/value_date/value_datetime/value_json` 任一非空。

### 5. extra_json 的合并更新
- `update_patient` 对 `extra_json` 走**深合并**而非替换：前端只提交修改过的脱敏字段时，未提交的键被保留。
- 嵌套 `merged_data` 子字典也单独合并。
- 注意：其它字段（`name / gender / department` 等）仍是覆盖语义。

## 与其他域的协作

### 文档与OCR
- 文档表 `document.patient_id` 是归档链接。
- 任何病例改动（创建/更新/删除）都触发 `DocumentService.invalidate_archive_tree_cache(owner_id)` 让前端"病例-文档归档树"重新拉取。

### AI抽取
- 病例详情提供入口 `POST /patients/{id}/ehr/update-folder` 一键抽取病例下所有可抽文档，详见 [[业务流程-新建病例]] 的下游与 [[AI抽取/业务概述]]。
- 病例 rail 上"病例夹更新中"指示器：`POST /patients/ehr-extraction-status` 批量查询活跃抽取任务。

### Schema模板与CRF
- 病例无直接绑定模板；模板通过 `data_context.schema_version_id` 间接挂上。
- `ehr` 域的 Schema 决定本病例 EHR Tab 的表单形态。

### 科研项目与数据集
- 病例侧不直接选项目，由项目侧调用 `POST /research/projects/{id}/patients` 纳入。
- 病例详情通过 `list_patient_projects` 反查关联项目。
