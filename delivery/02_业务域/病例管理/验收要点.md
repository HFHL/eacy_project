# 验收要点

> 本清单覆盖 [[业务流程-新建病例]] 与 [[业务流程-病例查询与档案查看]] 的可执行验收。`test_patient_api.py::test_patient_crud_flow` 已覆盖 CRUD 基础链路；以下场景需手工或扩展自动化验证。

## 通用前置数据

- 一个已登录用户（账号 A），其 `owner_id` 已能从 JWT 解析
- 另一个登录用户（账号 B），用于隔离性验证
- 至少一个已发布（`status=published`）的 `ehr` 域 SchemaTemplateVersion

---

## 用例 1：创建病例自动初始化 EHR 上下文

**步骤**
1. 账号 A 登录，`POST /api/v1/patients`，body 仅含 `{ "name": "验收-1" }`
2. 在数据库查询 `data_context` 表

**通过标准**
- [ ] 接口返回 201，响应体含 `id`
- [ ] 新建的 `Patient.owner_id` 等于账号 A 的 user id
- [ ] `data_context` 中存在一行 `patient_id = <new id>` 且 `context_type = patient_ehr`
- [ ] 该上下文的 `schema_version_id` = 当前 ehr 域最新已发布版本

**不通过的处理**：检查 `SchemaService.get_latest_published("ehr")` 是否返回正确版本；事务是否被异常吞掉（看后端日志）。

---

## 用例 2：owner_id 隔离

**步骤**
1. 账号 A 创建病例 P
2. 账号 B 登录，`GET /api/v1/patients/{P.id}`
3. 账号 B 调用 `GET /api/v1/patients`

**通过标准**
- [ ] 步骤 2 返回 404（不是 403、不是 200）
- [ ] 步骤 3 返回的 `items` 列表不含 P

---

## 用例 3：列表筛选与分页

**步骤**
1. 账号 A 创建三个病例：`["心衰张三", "高血压李四", "糖尿病王五"]`，科室分别为"心内科 / 心内科 / 内分泌科"
2. `GET /patients?keyword=心&page=1&page_size=10`
3. `GET /patients?department=心内科`
4. `GET /patients?keyword=不存在的字符串`

**通过标准**
- [ ] 步骤 2 返回 2 条（命中"心衰张三"——姓名含心；"高血压李四"——若 doctor_name 或诊断含心；准确数量取决于种子数据，需验证 ILIKE 跨三列联合匹配生效）
- [ ] 步骤 3 仅返回科室为"心内科"的两条
- [ ] 步骤 4 返回 `total=0`，`statistics.total_documents=0`

> [!info] 关键字匹配的三列
> `name / main_diagnosis / doctor_name`，OR 联合。本用例需要在种子数据中显式覆盖任一列含"心"才能验证完整。

## 用例 4：extra_json 合并语义

**步骤**
1. 账号 A 创建病例并设置 `extra_json = {"phone": "1380000", "merged_data": {"address": "北京"}}`
2. `PATCH /patients/{id}`，body `{"extra_json": {"id_card": "330***", "merged_data": {"city": "杭州"}}}`
3. `GET /patients/{id}` 查看

**通过标准**
- [ ] `extra_json.phone` 仍为 `"1380000"`（未被覆盖）
- [ ] `extra_json.id_card` 为 `"330***"`（新增）
- [ ] `extra_json.merged_data` 含 `{ "address": "北京", "city": "杭州" }`（嵌套合并）

---

## 用例 5：软删除级联

**步骤**
1. 账号 A 创建病例 P，给 P 上传 1 个文档，并通过项目 API 把 P 纳入某科研项目
2. `DELETE /patients/{P.id}`
3. 检查数据库

**通过标准**
- [ ] 接口返回 204
- [ ] `patients.deleted_at` 非空
- [ ] 该病例的 `document.deleted_at` 非空（软删）
- [ ] `project_patient` 中对应行 `status` 已被 `withdraw_by_patient` 更新为撤回态（具体值待与团队确认，参考 [[科研项目与数据集/README]]）
- [ ] `GET /patients/{P.id}` 返回 404
- [ ] `field_current_value / field_value_event` 等历史数据**未被删除**（审计链保留）

---

## 用例 6：数据完整度计算

**步骤**
1. 账号 A 创建病例 P，自动绑定 ehr Schema（假设叶子字段 10 个）
2. 通过 `PATCH /patients/{id}/ehr/fields/{path}` 填入 3 个字段值（其中一个是 `medications.0.name` 嵌套）
3. `GET /patients` 取回 P

**通过标准**
- [ ] `data_completeness` 约为 `3/10*100 = 30.0`
- [ ] 若额外填了 `medications.1.name`，完整度**不增加**（因为规范化后是同一路径 `medications.name`）
- [ ] `document_count` 等于实际未软删文档数

## 用例 7：活跃抽取状态批量查询

**步骤**
1. 对账号 A 的两个病例 P1、P2 各发起一次 `update-folder` 抽取（P1 触发后立即查询，P2 让其完成）
2. `POST /patients/ehr-extraction-status` body `{"patient_ids": ["P1.id", "P2.id"]}`

**通过标准**
- [ ] 返回的 `items` 长度 = 2
- [ ] P1 的 `active = true`，`latest_status ∈ {pending, running}`
- [ ] P2 的 `active = false`，`job_count = 0` 或反映已完成（具体语义见 `list_active_ehr_status_by_patients`，待与团队确认）

---

## 用例 8：创建病例失效归档树缓存

**步骤**
1. 账号 A 调用前端"文档归档树"接口缓存数据（具体接口见 [[文档与OCR/README]]）
2. 账号 A 创建一个新病例
3. 再次拉归档树

**通过标准**
- [ ] 第二次拉取应反映新病例的存在（即缓存被刷新）
- [ ] 仅 owner = A 的缓存被失效；账号 B 的缓存（如有）不受影响

## 不通过的处理

- API 行为偏差：对照 `backend/app/api/v1/patients/router.py` 与 `patient_service.py`，并参考 `tests/app/test_patient_api.py::test_patient_crud_flow` 的最小可复现路径
- owner 隔离失效：复核 `PatientRepository.get_active_by_id / list_active` 是否带 `owner_id` 过滤
- 自动化扩展：建议在 `tests/app/test_patient_api.py` 中补充隔离、过滤、合并三类场景的覆盖
