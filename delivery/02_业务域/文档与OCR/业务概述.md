# 业务概述

## 一句话定位

文档与 OCR 域把"用户上传的一个 PDF/图片"沿两条异步流水线（**OCR** → **Metadata**）转化为"可抽取的结构化原料"：版面文字 + 坐标 + 文档类型。文件本体永远落在对象存储，数据库只保留元数据与解析产物。

## 核心概念

### Document（文档）
单一表 `documents` 承载从上传到归档的全部生命周期。字段定义见 [[表-document]]，**不在此罗列**。关键字段语义：
- **`storage_provider` + `storage_path` + `file_url`**：原始文件在对象存储中的位置；`file_url` 是公网可读地址（如果开启），`storage_path` 是 OSS Object Key。
- **`status`**：业务总状态（见下文生命周期）。
- **`ocr_status` / `meta_status`**：两条独立流水线的子状态，互不阻塞。
- **`ocr_payload_json` / `parsed_data`**：归一化后的 OCR 产物（同源，详见 [[关键设计-OCR坐标归一化]]）。
- **`metadata_json`**：Metadata Agent 识别出的"患者姓名 / 文档类型 / 唯一标识符 …"等。
- **`doc_type` / `doc_subtype` / `doc_title` / `effective_at`**：从 `metadata_json` 提炼出的可查询字段。
- **`patient_id`**：归档目标病例；`null` 表示**未归档**。
- **`uploaded_by`**：与 [[病例管理/业务概述]] 一致，按用户隔离可见性。

### 文档生命周期（status 总状态机）

```text
                        upload_document()
                              │
                              ▼
        ┌─────────────────── uploaded ──── (patient_id != null) ──┐
        │                                                          ▼
        │            (DOCUMENT_OCR_AUTO_ENQUEUE=true)          archived
        ▼                                                          ▲
   ocr_pending ──── process_document_ocr ──┬── ocr_completed ──────┤
        ▲                                  │                       │
        │                                  └── failed              │
        │                                                          │
        └──── trigger_document_ocr (重抽) ◀──────── unarchive ◀────┘
```

- `uploaded`：未开 OCR 自动入队时的初态。
- `ocr_pending`：已入 OCR 队列，等待 Worker 拾取。
- `ocr_completed`：OCR 完成、尚未归档到病例。
- `archived`：已归档到某 `patient_id`；之后即使重抽 OCR，状态依然回到 `archived`（见 `process_document_ocr` 中的 `keep_archived` 分支）。
- `failed`：OCR 失败且文档尚未归档；已归档文档失败时 `status` 保持 `archived`、`ocr_status=failed`。
- `deleted`：软删除，仓库查询默认过滤。

> [!info] 两条子状态独立演进
> `ocr_status` 与 `meta_status` 是**并行流水线**：Metadata 依赖 OCR 完成，但 OCR 失败不会把 `meta_status` 拉回；OCR 重抽也不会重置 Metadata。前端"待解析"徽标的判断在 `is_pending_process_document`（见 [[关键设计-未归档池与归档]]）。

### 三种"待办池"（左侧目录三档徽标）
档案树 `GET /documents/v2/tree` 返回的 `counts` 把可见文档分为互斥三档：
- **`parse_total`** —— OCR 或 Metadata 未完成（`is_pending_process_document`）。
- **`todo_total`** —— 已解析但仍待归档（进入 `todo_groups`）。
- **`archived_total`** —— 已归档。
三者之和等于总文档数，避免同一份文档同时计入两档。

## 对象存储与本地存储

EACY 当前只支持一种 storage backend：**阿里云 OSS**。`backend/app/storage/document_storage.py` 的 `build_document_storage()` 在 `DOCUMENT_STORAGE_PROVIDER != "oss"` 时直接抛 `RuntimeError("EACY document storage is OSS-only.")`。

- 上传时 `AliyunOssDocumentStorage.save` 算 SHA-256、PUT 到 `documents/YYYY/MM/<uuid><ext>`，并把结果 `(provider, path, url, size, sha256)` 写入文档行。
- 预览时统一签发**短期有效**签名 URL（`get_signed_url`，默认 1 小时，最长 24 小时），前端通过 `/documents/{id}/preview-url` 或 `/documents/{id}/stream` 拿到。
- 流式接口 `stream_document` 走后端反向代理：用 `httpx.AsyncClient` 拉 OSS、流式回吐给浏览器，避免把签名 URL 暴露到客户端历史。

> [!warning] 本地磁盘存储尚未实现
> 文档结构里曾提到"OSS / 本地"两种 backend，目前**只有 OSS**。若需要本地落盘（开发环境、离线部署），需要新增一个 `LocalDocumentStorage` 类并在 `build_document_storage` 注册。TBD。

## 与其他域的协作

### 病例管理
- 上传时若带 `patient_id`，文档**直接归档**到该病例（`status=archived`、`archived_at=utcnow()`）。
- 上传时不带 `patient_id`，进入未归档池，由用户在 FileList 页"待归档"分组里手动归档（见 [[关键设计-未归档池与归档]]）。
- 病例软删除会级联软删该病例下所有文档（详见 [[病例管理/业务概述]]）。

### AI 抽取
- OCR 完成后 `process_document_ocr` 自动触发两件事：
  1. `_enqueue_metadata_task` 入 Metadata 队列。
  2. `enqueue_ready_extraction_jobs(document_id)` 唤醒此前因"等待 OCR"而挂起的抽取任务（`wait_for_document_ready=False`）。
  3. 如果上述无任何唤醒任务且文档已绑定病例，则 `create_and_enqueue_patient_ehr_extraction` 直接为该病例新建一个 `patient_ehr` job。
- Metadata 完成后还会再调一次 `enqueue_ready_extraction_jobs`，覆盖"先等 Metadata 再抽"的链路。
- 文档作为字段证据时，删除前由 `GET /documents/{id}/evidence-impact` 提示影响范围；删除后字段值保留，证据来源被标记为"已删除文档"。

### Schema 模板与 CRF
- 抽取规划阶段会根据 `doc_type` / `doc_subtype` 决定走哪条 prompt 链路，详见 [[Schema模板与CRF/业务概述]]。Metadata 服务的 `doc_type` 是此处的输入。

### 管理后台
- OCR / Metadata 的 Celery 任务统一登记到 `async_task`，前端通过 [[管理后台/异步任务进度追踪]] 全局轮询。
