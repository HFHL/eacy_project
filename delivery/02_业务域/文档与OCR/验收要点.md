# 验收要点

## 验收范围
覆盖文档与 OCR 域的核心链路：上传 → 落 OSS → OCR → Metadata → 未归档分组 → 归档 → 失败重抽。深入业务规则参见 [[业务概述]] 及各业务流程文档。

## 前置数据
- 一名已登录用户，账号下没有同名文件。
- 至少 2 张测试 PDF：A 包含可识别患者信息（姓名 + 身份证号），B 内容空白或全图无文字。
- 一个已存在病例 P1（属于当前用户），姓名与 PDF A 的 Metadata 匹配。
- 配置：`DOCUMENT_OCR_AUTO_ENQUEUE=true`、OSS / TextIn / LLM 凭据齐全。

## 验收步骤与通过标准

### 用例 1：未带 patient_id 上传 → 自动 OCR 串到 Metadata
1. `POST /documents`，上传 PDF A，不带 `patient_id`。
2. 立刻 `GET /documents/{id}`：`status=ocr_pending`、`ocr_status=queued`、`storage_provider=oss`、`storage_path` 形如 `documents/YYYY/MM/<uuid>.pdf`。
3. 60s 内轮询，`ocr_status` 经历 `running → completed`，`parsed_data.pages.length > 0`、`ocr_text` 非空。
4. 进一步 30s 内 `meta_status=completed`，`doc_type` / `metadata_json.result.患者姓名` 非空。

- [ ] OCR 自动入队
- [ ] OCR 完成后 Metadata 自动入队
- [ ] 归一化产物可在 `ocr_payload_json.lines/blocks/tables` 看到

### 用例 2：带 patient_id 上传 → 直接归档
1. `POST /documents`，PDF A + `patient_id=P1`。
2. `GET /documents/{id}`：`status=archived`、`archived_at` 非空、`patient_id=P1`。
3. 调用 `GET /documents/v2/tree`，`counts.archived_total` 比上传前 +1。

- [ ] 直接归档不经过待归档分组
- [ ] archive_tree 缓存被失效（数值实时刷新）

### 用例 3：OCR 失败 → 状态机正确
1. 临时把 `TEXTIN_API_URL` 改成不可达地址（或断网）。
2. `POST /documents/{id}/ocr` 重抽 PDF A。
3. 任务结束后：`ocr_status=failed`，`ocr_payload_json.errors[0].message` 含 HTTP / 超时类描述。
4. 修复后再次 `POST /documents/{id}/ocr`，最终 `ocr_status=completed`。

- [ ] 失败状态显式落库
- [ ] 重抽幂等：连发两次时第二次返回 409，不会同时 `running`

### 用例 4：空白文档的 Metadata skip
1. 上传 PDF B（无可识别文本）。
2. OCR 完成后 `ocr_text` 为空。
3. `POST /documents/{id}/metadata`。
4. `meta_status=skipped`，`metadata_json.error.message` 含 "OCR text is empty"。

- [ ] skipped 是终态，不会重试
- [ ] FileList 不把此类文档算入 `parse_total`（注：当前实现下 OCR 已完成且 metadata_json 为空仍算待解析，TBD 业务期望）

### 用例 5：未归档分组 → 候选病例 → 一键归档
1. 上传 3 份内容相同患者的 PDF（不带 `patient_id`），等 OCR + Metadata 完成。
2. `GET /documents/v2/tree`：3 份被聚成 1 个组，`status_set` 包含 `auto_archived` 或 `pending_confirm_review`。
3. `GET /documents/v2/groups/{group_id}/documents` 看到 3 份文档与候选 P1。
4. `POST /documents/v2/groups/{group_id}/confirm-archive?patient_id=P1`，返回 `archived_count=3`。
5. 3 份文档 `status=archived` 且 `patient_id=P1`；`todo_total` 减少 3。

- [ ] 并查集按"唯一标识符 + 弱信息打分 ≥ 5"工作
- [ ] 整组归档原子可见（缓存失效后 tree 立即更新）

### 用例 6：删除前的证据影响提示
1. 对一份已被 AI 抽取使用的文档调 `GET /documents/{id}/evidence-impact`。
2. 返回 `evidence_count > 0` 且 `fields[]` 列出受影响字段 `code/title`。
3. 调 `DELETE /documents/{id}`：成功；查询相关字段值仍存在，但 `field_value_evidence` 显示来源已断链。

- [ ] 删除前能拿到完整影响清单
- [ ] 字段值本身不被级联删

### 用例 7：跨用户隔离
1. 用户 A 上传 PDF C。
2. 用 用户 B 调 `GET /documents/{C}`：返回 404。
3. 用户 B 调 `GET /documents/v2/tree`：看不到 PDF C 的计数。

- [ ] `uploaded_by` 强隔离生效（与 [[病例管理/业务概述]] 一致）

### 用例 8：预览签名 URL 不可滥用
1. `GET /documents/{id}/preview-url?expires_in=60`，拿到 `temp_url`。
2. 5 秒内访问可下载；超过 60s 后访问应 403 / SignatureDoesNotMatch（OSS 侧）。
3. 同样的 URL 在另一台浏览器也能用（**当前实现是签名 URL，没有 IP 绑定**，TBD 是否需要加强）。

- [ ] expires_in 上限 86400 由后端强制
- [ ] 安全注意点已在文档中明示

## 不通过的处理
- OSS / TextIn / LLM 凭据问题 → 找运维核对 env，参考 [[业务流程-文档上传与存储]] 与 [[业务流程-OCR处理]] 的"异常分支"表。
- 状态机异常（如卡在 `running`） → 查 `async_task` 是否丢消息；手动 `POST /documents/{id}/ocr` 重抽。
- 分组聚错 → 检查 `metadata_json.result.唯一标识符` 是否被 LLM 正确识别；必要时改 prompt 或调整弱信息门槛（见 [[关键设计-未归档池与归档]]）。
