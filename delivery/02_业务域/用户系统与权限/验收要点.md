# 验收要点

> 用户系统的最小验收清单。覆盖登录态、token 生命周期、`ENABLE_AUTH` 开关、admin 守门、数据作用域。

## 1. 登录基本路径

- [ ] `POST /api/v1/auth/register` 用合法邮箱密码注册 → 201，响应含 `access_token / refresh_token / user`。
- [ ] 重复邮箱注册 → 409。
- [ ] `POST /api/v1/auth/login` 用刚注册的账号 → 200，`user.last_login_at` 更新。
- [ ] 错密码 / 不存在邮箱 / `is_active=false` → 统一 401（**不区分原因**，避免账号枚举）。

## 2. JWT 生命周期

- [ ] 用 `access_token` 调 `GET /api/v1/auth/me` → 返回当前用户的 `id/username/role/permissions`。
- [ ] 等到 access_token 过期或人为修改使其失效 → 业务接口首次返回 401 → 前端 `request.js` 自动调 `/auth/refresh` → 重放成功（用户无感）。
- [ ] refresh_token 也失效 → 自动跳 `/login` 且 `localStorage` 中三件套（access/refresh/user_info）被清掉。
- [ ] 并发 N 个业务请求同时遇到 401 → `/auth/refresh` 只被实际调用 1 次（`refreshPromise` 单例）。

## 3. `ENABLE_AUTH` 开关行为

> 参考 `backend/tests/app/test_auth_dev_mode.py`。

- [ ] `ENABLE_AUTH=false`（默认）：**不带 Authorization 头**访问任意业务接口（如 `GET /api/v1/auth/me`、`GET /api/v1/patients`）→ 放行，`CurrentUser` 为 `dev_admin`。
- [ ] `ENABLE_AUTH=true`：同样的不带头请求 → 401 `Missing authorization token`。
- [ ] `ENABLE_AUTH=true` 且带**伪造签名**的 token → 401 `Invalid authorization token`。

## 4. Admin 守门

- [ ] 普通 `role="user"` 账号调 `GET /api/v1/admin/stats` → 403 `Admin permission required`。
- [ ] 把账号 `role` 改为 `"admin"`（或在 token payload 注入 `permissions=["*"]`）→ 同接口 200。

## 5. 数据作用域（owner_id / uploaded_by）

> 这是当前实际生效的"用户只能看到自己数据"机制，必须逐接口验。

- [ ] 用户 A 登录创建 1 个病例 + 上传 1 个文档；用户 B 登录调 `GET /api/v1/patients` / `GET /api/v1/documents` → **看不到 A 的数据**。
- [ ] 用户 B 直接以 A 的 `patient_id` 访问 `GET /api/v1/patients/{id}` → 404（仓储层加了 `owner_id` 过滤）而非 200。
- [ ] `ENABLE_AUTH=false` 开发期 `dev_admin` 调上述接口 → 能看到所有用户的数据（`uuid_user_id_or_none` 返回 `None` → 不过滤），用于联调。
