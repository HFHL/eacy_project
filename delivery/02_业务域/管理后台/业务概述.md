# 业务概述

> [!info] 一句话定位
> 管理后台是**全平台的运维观测面板**：只读聚合 async_task / 用户 / 项目 / 模板 / 文档，并对单个抽取任务提供完整下钻链路（批次 → 子任务 → ExtractionRun → 字段证据 → LLM 调用）。

## 覆盖范围

管理后台**当前**承担三个职责：

### 1. 异步任务监控（核心）

- 列表视图：按 `task_type`（`project_crf` / `patient_ehr` / `targeted` 等）和 `status`（运行 / 等待 / 完成 / 失败 / 停滞 / 取消）筛选 [[表-async_task]] 中的批次
- 详情视图：选中一条批次后展开
  - 批次 summary：项目 / 患者 / 模板 / 目标表单 / 进度计数
  - 文档级 jobs：每个 [[表-extraction_job]] 的最新 [[表-extraction_run]]、模型、提示词版本、字段候选数、证据数
  - 已抽取字段表：复用 [[AI抽取/证据归因机制]] 的 field_value_event + field_value_evidence
  - LLM 调用明细：来自 `extraction_run.parsed_output_json.validation_log`
- 实时进度：`running` / `pending` 状态的任务接 SSE（`useExtractionProgressSSE`），其它状态走列表 5s 轻量轮询

### 2. 用户与项目概览

- 用户列表（来自 `users` 表）：用于查看注册情况与最后登录，不含编辑能力
- 项目列表（来自 `research_projects` + `project_patients` 聚合患者数）：只读
- 模板列表（来自 `schema_templates` + 最新 `schema_template_versions`）：只读，字段数由 `schema_json` 递归计数

### 3. 队列健康度（部分）

- 顶部 `overview` 卡片：用户 / 患者 / 文档 / 项目 / 模板总数 + `active_tasks`（处于 `pending / queued / running / stale` 的任务总和）
- 停滞探测：`AdminTaskService._is_stale` 用 `ADMIN_TASK_STALE_AFTER = 15min` 阈值，对 `running / queued` 状态但 `heartbeat_at` 超期的任务标记为 `stale`
- 真正的队列长度、Worker 存活、Redis 健康度等运维信息**当前不在管理后台内**，TBD（见 [[Celery任务运维]]，待写）

## 不做的事

- **不写业务数据**：所有 `/admin/*` 路由当前只暴露 GET；无编辑、无删除
- **不取代业务页面**：病例归档、字段审核、项目纳入仍在各业务页面完成
- **不替代日志系统**：诊断信息以 `async_task_events` 为主，应用日志 / Celery 日志见 [[部署运维]]（待写）
- **不直接触发重试**：当前前端没有"重试"按钮调用（参见 [[业务流程-异步任务监控]] 的"重试触发"段，TBD）

## 关键设计原则

> [!example] 三个原则
> - **观测层 vs 业务层分离**：`async_task_*` 三张表是观测层，不替代 `extraction_jobs / documents` 这些业务表；状态可能滞后但不影响业务正确性
> - **状态归一化**：业务表的状态字（`created` / `succeeded` 等）在 API 层映射为前端友好的统一枚举（`pending` / `completed`），由 `_normalize_batch_status` / `_normalize_item_status` 完成
> - **聚合在服务层**：admin 列表的 `type_counts` / `status_counts` 在 Python 层算，不依赖 SQL group by；规模假设 ≤ 1000 条/列表，超出需要分页改造（TBD）
