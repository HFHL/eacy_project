# 验收要点

> 本文档列出 AI 抽取域上线/验收时必须可手动复现的关键用例。每条用例都标明触发入口、预期行为、可观察证据（哪些表/接口能看出来）。

## 用例 1：单文档抽取端到端可用

**触发**：在 `PatientDetail/DocumentsTab` 选一份已 OCR 的"出院记录" → 触发"抽取"。

**期望**：
- 1 个 `ExtractionJob` 入队，5~30 秒内变 `completed`
- `extraction_runs` 表新增一行 `run_no=1, status=completed`
- 该文档对应 form_key 下的字段在 EhrTab 立即可见

**验证查询**：
```sql
SELECT status, progress, error_message FROM extraction_jobs WHERE id = :job_id;
SELECT count(*) FROM field_current_values WHERE context_id = :ctx;
SELECT count(*) FROM field_value_evidence WHERE document_id = :doc;
```

## 用例 2：证据归因正确性（必查）

**触发**：随机抽 3 个抽取出来的字段值，点开"证据"。

**期望**：
- 每个字段至少 1 条 evidence
- `quote_text` 必须是该文档 OCR 文本的真子串（去空白后包含）
- 至少 80% 的 evidence 有 `page_no` + `bbox_json.polygon`
- 缺位置的 evidence 必须带 `fallback_strategy=sibling_field_location` 标识

**验证查询**：
```sql
SELECT e.quote_text, e.page_no, e.bbox_json
FROM field_value_evidence e
JOIN field_value_events ev ON ev.id = e.value_event_id
WHERE ev.context_id = :ctx LIMIT 20;
```

> 见 [[关键设计-证据归因机制]]。

## 用例 3：字段值历史可见

**触发**：对某字段先做 AI 抽取，再人工修改值，再次 AI 抽取（不同文档）。

**期望**：
- `field_value_events` 累计 3 行（event_type: ai_extracted / manual_edit / ai_extracted）
- `field_current_values` 只有 1 行，且 `selected_event_id` 指向最新一条
- 调 `GET /patients/{id}/fields/{path}/candidates` 返回 3 个候选，`has_value_conflict=true`（值不同时）

> 见 [[关键设计-字段值历史与变更链]]。

## 用例 4：批量抽取进度可见

**触发**：病例下有 10 份已 OCR 文档，点"更新电子病历夹"。

**期望**：
- `async_task_batches` 立即新增 1 行 `task_type=patient_ehr_folder_extract`
- `async_task_items` 新增 N 行（N = `created_jobs`）
- 前端 `globalBackgroundTaskPoller` 在 2~3s 内显示批次条目，progress 单调递增
- 全部完成后 batch.status 切到 `succeeded` 或 `completed_with_errors`

**验证查询**：
```sql
SELECT status, progress, total_items, succeeded_items, failed_items
FROM async_task_batches WHERE id = :batch;
```

> 见 [[业务流程-病例EHR批量更新]] 与 [[关键设计-异步任务进度追踪]]。

## 用例 5：抽取失败可重试

**触发**：人为制造 LLM 失败（断网或 OPENAI_API_KEY 错误）→ 提交一个抽取 Job。

**期望**：
- Celery 自动重试 3 次（每次写 `async_task_event` 一条 error）
- 3 次失败后 Job=failed，`error_message` 非空
- 前端在该 Job 上显示"重试"按钮
- 修好网络/Key 后，点 `/retry` → 新建 `run_no=2` → completed
- `extraction_runs` 表保留 1 失败 + 1 成功的两行

> 见 [[业务流程-抽取任务生命周期]]。

## 用例 6：嵌套字段多实例

**触发**：上传"出院带药"清单 PDF，含 3 条用药记录，触发用药记录 form 抽取。

**期望**：
- LLM 输出 `records[0].record` 为 3 条数组
- 落库后 `field_current_values` 中可见 `field_path` 以 `.0.` / `.1.` / `.2.` 区分
- 调 `GET /patients/{id}/ehr` 返回时前端能看到 3 行用药记录

> [!info] 当前实现限制（见 [[关键设计-嵌套字段与RecordInstance]] TBD）
> 这 3 条不会自动建 3 个 RecordInstance，靠 field_path 末端下标区分。

## 用例 7：抽取规划只匹 primary

**触发**：病例下有"出院记录"和"诊断证明"两份文档；schema 中"用药记录" form 只把"出院记录"列为 primary、"诊断证明"列为 secondary；触发批量更新文件夹。

**期望**：
- 只有"出院记录"被规划成 `targeted_schema` Job 用于"用药记录" form
- "诊断证明"在响应 `skipped` 列表里或被规划到其它 form
- 不会把"诊断证明"无关内容塞进"用药记录"

> 见 [[业务流程-Schema字段规划]] 与 [[业务流程-病例EHR批量更新]]。

## 用例 8：完成态保护

**触发**：对一个已 `completed` 的 Job 调 `POST /cancel` 与 `DELETE /{id}`。

**期望**：
- `/cancel` 返回 409 `Completed extraction job cannot be cancelled`
- `DELETE` 返回 409 `Extraction job has runs or field events and cannot be deleted`
- `/retry` 返回 409 `Failed extraction job must be retried`（因状态不是 failed）

> 见 [[业务流程-抽取任务生命周期]] 的 `_ensure_can_retry` / `_ensure_can_process`。
