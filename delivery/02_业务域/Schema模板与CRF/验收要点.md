# 验收要点

> 本模块的可执行验收用例。每条用例都应能用真实接口 / 真实页面跑通。

## 用例

### A1. 模板设计与发布（端到端）
- **前置**：登录、具备模板管理权限。
- **步骤**：
  1. 进入 `CRFDesigner`（新建路径），填写模板名/分类/描述。
  2. 拖入若干字段（含 `radio` 枚举、`date`、`table` 多行），保存草稿。
  3. 在模板列表确认状态为 `draft`。
  4. 点"发布模版"。
- **期望**：
  - [ ] `schema_template` 新增一行，`template_code` 自动生成且全局唯一。
  - [ ] `schema_template_version` 新增 `version_no=1`、`status=published`、`published_at` 非空。

### A2. 版本不变性
- **前置**：A1 的版本已发布。
- **步骤**：
  1. 用 `POST /schema-templates/{id}/versions` 再发一次 `version_no=2` draft。
  2. 直接尝试通过任何接口修改 v1 的 `schema_json`。
- **期望**：
  - [ ] 没有任何 HTTP 接口允许修改已发布版本的 `schema_json`（路由检查）。
  - [ ] v2 与 v1 并存，互不影响。

### A3. 病例自动绑定最新 ehr published
- **前置**：`template_type='ehr'` 已有至少一个 published 版本（v1）。
- **步骤**：
  1. 新建病例 P1，记下其 `data_context.schema_version_id`。
  2. 发布 ehr v2。
  3. 新建病例 P2。
- **期望**：
  - [ ] P1 的 `schema_version_id` 仍指向 v1。
  - [ ] P2 的 `schema_version_id` 指向 v2。

### A4. 项目绑定指定版本
- **步骤**：
  1. 创建科研项目，绑定 crf 模板的 v1 版本。
  2. 之后 crf 模板再发 v2 published。
  3. 触发项目数据集导出。
- **期望**：
  - [ ] `project_template_binding.schema_version_id` 仍为 v1。
  - [ ] 导出文件的结构按 v1 的 schema 聚合，不含 v2 新增字段。

### A5. 已引用 draft 不可删
- **步骤**：
  1. 手工构造一条 `project_template_binding` 引用某 draft 版本（或测试场景里把 draft 暂时绑给一个测试项目）。
  2. 调 `DELETE /schema-template-versions/{vid}`。
- **期望**：
  - [ ] 返回 409 Conflict，错误信息提示有引用。

### A6. published 删除即软删
- **步骤**：调 `DELETE /schema-template-versions/{vid}`，其中 vid 为 published 状态。
- **期望**：
  - [ ] 200/204；记录仍存在，`status` 变 `deprecated`。
  - [ ] 已引用该版本的病例/项目继续可用，schema_json 内容未被改动。

### A7. form_key 路径与 CategoryTree 一致
- **前置**：任一已发布 schema 含 `基本信息.人口学情况` 这样的二级结构。
- **步骤**：
  1. 调 `schema_field_planner.schema_top_level_forms` 拿 form_key 列表。
  2. 在前端 `SchemaForm` 渲染同一 schema，对照 CategoryTree 第二层节点。
- **期望**：
  - [ ] 两边的 form_key 集合完全一致，包括"无 properties 兜底为文件夹名"的情况。

### A8. 模板归档不影响老引用
- **步骤**：把某 ehr 模板的 `status` 改 `archived`。
- **期望**：
  - [ ] `get_latest_published('ehr')` 不再返回该模板的版本。
  - [ ] 已绑定其历史版本的病例 / 项目仍能正常打开、抽取、导出。
